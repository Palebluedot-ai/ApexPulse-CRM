# M1 到 M5 产品路线图

## 1. 目的

这份文档回答四个问题：

- 当前产品最终要做到什么程度
- M1.15 到 M2 是否已经足够清楚
- M3、M4、M5 是否应该现在就写出来
- 能不能一次性把后续全部开发完

这份文档是自包含文档。

阅读这份文档不需要依赖其他文档。

## 2. 当前结论

当前产品计划没有完整覆盖到 M5 的可执行细节。

当前已经足够清楚、可以继续开发的是：

- M1.15 Weekly report / 本周 to-do list
- M1.16 图片预览体验 polish

当前有方向、但不能直接开发的是：

- M2 Leader / 下级关系
- M2 OCR / Vision API

当前只适合做路线图、不适合直接开发的是：

- M3 AI 辅助提取和跟进建议
- M4 部署、备份、运维和长期稳定性
- M5 多人团队产品化和规模化

不能一次性把 M1.15 到 M5 全部开发完。

可以一次性写出 M1 到 M5 的路线图。

可以一次性写出 M2 的澄清问题清单。

不能一次性写出 M2 到 M5 的最终代码。

原因：

- M2 涉及权限边界，错误设计后面很难改
- M3 涉及 AI provider、成本、准确率和人工 Review 边界
- M4 涉及部署位置、备份策略、密钥管理和数据迁移
- M5 涉及是否真的做成多人团队产品，不是当前个人 dogfood 的自然延伸

## 3. 里程碑定义

### 3.1 M1：个人可用闭环

M1 的目标是让 Chao 自己可以稳定使用。

M1 的核心闭环是：

```text
录入证据 -> Review 确认 -> 写入客户 -> 更新跟进状态 -> 生成任务 -> 本周复盘
```

M1 完成后，系统应该做到：

- 可以登录
- 可以录入文字
- 可以上传图片
- 可以保留原始证据
- 可以 Review 自然字段
- 可以确认入库
- 可以自动创建新客户
- 可以自动生成 followup 任务
- 可以看到客户列表
- 可以看到客户详情第一屏
- 可以完成和恢复任务
- 可以看到本周 report 和本周 to-do list
- 可以在本地 Mac mini 或本地开发环境稳定跑

M1 不做：

- 下级管理
- 团队权限
- 自动跳过 Review
- OCR 自动识别
- Vision API
- 云部署强绑定
- 复杂客户分层
- 客户合并

M1 建议终点：

```text
M1.16
```

M1.16 完成后，个人 dogfood 阶段应该可以暂停加功能，进入真实使用和问题收集。

### 3.2 M2：团队和智能录入入口

M2 的目标不是简单加功能。

M2 是两个大方向的分叉：

- 团队协作
- OCR / Vision API

这两个方向都需要额外澄清。

M2A：Leader / 下级关系。

M2A 目标：

- Leader 可以添加下级
- Leader 可以查看下级跟进情况
- 下级默认只看到自己负责的数据

M2A 开发前必须确认：

- 下级能不能看到 Leader 的客户
- 平级之间是否完全隔离
- Leader 能不能编辑下级客户
- Leader 能不能完成下级任务
- 下级离职或转组时客户归属如何处理
- Report 是个人 report 还是团队 report
- Manager 链条是否只有一层

M2B：OCR / Vision API。

M2B 目标：

- 上传截图或名片照片后，系统辅助提取客户、公司、需求、下一步
- AI 提取结果仍然进入 Review
- 人必须确认后才能入库

M2B 开发前必须确认：

- 用哪个 API provider
- API key 怎么存
- 每月成本上限是多少
- 截图是否允许发送到外部 API
- 识别失败时如何提示用户
- 是否保留纯手工录入

M2 不允许：

- 默认让 AI 自动入库
- 默认让 Leader 看到所有数据但没有测试
- 只做 UI，不做数据可见性测试
- 把真实客户截图发到外部 API 测试

### 3.3 M3：AI 辅助跟进系统

M3 的目标是让系统从“记录和提醒”进入“辅助判断”。

M3 可以包括：

- AI 生成沟通摘要
- AI 生成下一步建议
- AI 标记客户风险
- AI 生成 follow-up message 草稿
- AI 生成 weekly report 文字版

M3 开发前必须确认：

- 哪些内容可以让 AI 自动写
- 哪些内容只能作为建议
- 用户是否需要看到 AI 置信度
- 错误建议如何回滚
- 成本是否可控

M3 不应该在 M1 或 M2 里顺手做。

原因：

AI 建议会影响销售判断，不能在数据闭环还没稳定时提前加。

### 3.4 M4：部署、备份和长期稳定性

M4 的目标是让系统从“本地可用”变成“长期可靠”。

M4 可以包括：

- Vercel 部署
- Supabase 或 Postgres 云数据库
- 文件存储迁移
- 数据备份
- 登录安全升级
- 环境变量管理
- 错误日志
- 基础监控
- PWA 安装体验

M4 开发前必须确认：

- 继续 Mac mini 本地部署，还是迁云
- 云数据库选 Supabase 还是其他 Postgres
- 图片附件放本地、S3、Supabase Storage 还是其他存储
- 是否需要公司服务器部署
- 数据备份频率
- 密钥管理方式

M4 不应该在 M1 里做。

原因：

当前最重要的是产品闭环，不是过早工程化部署。

### 3.5 M5：团队产品化

M5 的目标是从 Chao 自己使用，变成一个小团队可以长期使用的 CRM。

M5 可以包括：

- 多用户 onboarding
- 团队 dashboard
- 成员绩效视图
- 客户分配
- 客户转移
- 数据导入导出
- 更完整的审计记录
- 更完整的权限矩阵
- 多渠道集成

M5 开发前必须确认：

- 这个系统是否真的要给团队长期使用
- 是内部工具，还是可以产品化成 side project
- 团队成员是否愿意使用
- 管理视图会不会造成额外沟通成本
- 什么指标真的能反映销售工作质量

M5 现在不应该直接开发。

原因：

M5 是组织流程问题，不只是代码问题。

## 5. 一次性开发判断

一次性开发 M1.15 到 M5 不合适。

可以这样推进：

1. 继续完成 M1.15。
2. 继续完成 M1.16。
3. M1 结束后做一次真实 dogfood。
4. 根据真实 dogfood 决定先做 M2A 团队权限，还是 M2B OCR / Vision API。
5. M2 开始前必须做一次 Office Hours 或需求澄清。
6. M2 完成后再决定 M3 是否值得做。
7. M4 和 M5 等真实使用稳定后再进入。

这样做的好处：

- 不会过早写死权限模型
- 不会过早接入外部 AI 成本
- 不会为了“完整路线图”牺牲当前可用性
- 每个阶段都能 dogfood
- 每个阶段都能回滚
