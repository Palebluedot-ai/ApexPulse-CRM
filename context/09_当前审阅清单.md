## 3. 建议 Chao 亲自审阅的 3 个点

### 3.1 客户合并规则

当前文档已经确认：

- 第一版以“人”为主对象
- 一个人可能有多个联系方式

但还没完全定死的是：

- 同一个人如果先来自名片、后来自聊天截图，第一版是否默认手工合并
- 系统是否允许先存在重复客户，再由用户后续整理

当前推荐默认值：

- 第一版允许先存在重复客户
- 不做自动合并
- 发现疑似重复时，后续只做提示，不自动改数据

当前选择：`确认采用默认值`

原因：

- 这更稳
- 误合并的代价比暂时重复更高

### 3.2 待确认队列的粒度

当前文档已经确认：

- 自动识别结果必须先进待确认队列
- 用户至少可以 `confirm / edit / skip`

但还没完全定死的是：

- 第一版是“整条事件确认”
- 还是“每个字段单独确认”

当前推荐默认值：

- 第一版按“整条事件确认 + 可编辑字段”处理
- 不做每个字段独立状态机

当前选择：`确认采用默认值`

原因：

- 更简单
- 更符合第一版先跑通闭环的目标
- 后面如果真需要更细粒度，再往字段级扩展

### 3.3 客户详情页头部字段

当前文档已经列了建议字段，但值得你快速过目一下，确认“这是不是你平时最想先看到的信息”。

当前推荐第一屏结构：

- 客户名
- 公司名
- 介绍来源标签
- 最近联系时间
- 当前跟进状态
- 下一次建议跟进时间
- 最新沟通卡片

最新沟通卡片包含：

- 最新沟通摘要
- 下一步 / 承诺
- 风险或待补充信息
- 1 到 3 张原始截图缩略图
- 完整原始依据入口

当前选择：`最新沟通摘要应放在第一屏，并参考 V0 保留截图和原始依据的展示方式`

## 5. 当前可继续推进的默认值

我可以按下面这组默认值继续推进 `Phase 1`：

- 允许重复客户暂存，不做自动合并
- 待确认按“整条事件确认 + 字段可编辑”处理
- 客户详情第一屏展示最新沟通卡片
- 最新沟通卡片保留摘要、截图缩略图和原始依据入口

这组默认值是当前最稳、最省返工的一条路。
