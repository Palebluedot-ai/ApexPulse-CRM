## 3. M1 目标

M1 不追求完整系统，而追求第一条垂直路径跑通：

1. 上传截图 / 输入备注 / 上传名片照片
2. 保存原始内容
3. 生成待确认结果
4. 用户确认
5. 生成客户事件
6. 刷新客户跟进状态
7. 生成或补充任务

如果这条链不能跑通，其他功能都只是外围装饰。

## 5. 开发阶段拆解

### Phase 0：工程骨架

目标：

- 建立可持续开发的项目结构

任务：

- 初始化 Next.js 15 项目
- 配置 Tailwind / shadcn/ui
- 配置 pnpm / lint / typecheck / test 基础能力
- 约定环境变量方式
- 预留本地部署方案

完成标准：

- 项目可以启动
- 基本检查命令可跑

---

### Phase 1：数据模型落地

目标：

- 把草案变成第一版真实 schema

任务：

- 建 `users`
- 建 `parties`
- 建 `events`
- 建 `attachments`
- 建 `tasks`
- 定义关键枚举
- 定义跟进状态相关字段

完成标准：

- 可以迁移数据库
- 类型定义稳定

---

### Phase 2：录入入口

目标：

- 让用户能真正把内容送进系统

任务：

- 首页或录入页 UI
- 图片上传
- 文字备注录入
- 名片照片录入
- 原始内容保存

完成标准：

- 用户可以提交真实内容
- 原始内容能持久化

---

### Phase 3：待确认闭环

目标：

- 打通“自动提取 -> 用户确认”的中间层

任务：

- 提取结果结构
- 待确认列表页
- 确认 / 修改 / 跳过动作
- review 状态流转

完成标准：

- 用户可处理待确认记录
- 记录确认后进入正式事件

---

### Phase 4：客户与时间线

目标：

- 让记录沉淀成可查看的客户档案

任务：

- 客户列表
- 搜索
- 客户详情页
- 时间线展示
- 原始内容预览

完成标准：

- 可以通过客户查看历史记录
- 可以通过搜索找到客户或记录

---

### Phase 5：跟进状态刷新

目标：

- 实现第一版最关键的产品价值

任务：

- 每次事件 confirmed 后回写 party
- 刷新 `last_contact_at`
- 刷新 `last_contact_summary`
- 刷新 `next_followup_at`
- 刷新 `followup_status`

完成标准：

- 新记录确认后，客户状态自动变化

---

### Phase 6：任务体系

目标：

- 让系统不仅能存，还能推动跟进

任务：

- 自动任务生成
- 手动新建任务
- 完成 / 恢复任务
- 首页或任务页展示待办

完成标准：

- 任务可以生成、查看、完成

---

## 6. 当前推荐顺序

建议严格按下面顺序推进，不要跳：

1. Phase 0 工程骨架
2. Phase 1 数据模型
3. Phase 2 录入入口
4. Phase 3 待确认闭环
5. Phase 4 客户与时间线
6. Phase 5 跟进状态刷新
7. Phase 6 任务体系

原因很简单：

- 没有录入入口，就没有数据
- 没有待确认闭环，自动识别就不可信
- 没有客户与时间线，就看不到积累效果
- 没有跟进刷新，就不是这个产品的核心价值
- 没有任务体系，就无法形成后续动作

## 10. 2026-05-26 当前下一步

当前 M1 基础闭环已经完成：

- 数据库连接和 seed
- 文字与图片录入 API
- 待确认队列 API
- confirm 后刷新客户字段
- 客户列表和客户详情第一屏
- 任务创建、展示、完成、恢复
- 客户列表第一轮 UI 密度校准

当前不建议继续扩数据表或新增大功能。

下一步应进入：

`M1.1：操作入口补齐`

目标：

- 让 Chao 不再依赖 curl 或 API 手工调用
- 可以从页面完成录入和待确认处理
- 让 M1 进入真正 dogfood 状态

优先开发顺序：

1. `/capture` 录入页面
2. `/review` 待确认队列页面
3. 首页入口调整，把“录入”和“待确认”放到更明显位置

### 10.1 `/capture` 录入页面

目标：

- 输入文字备注
- 录入图片证据元数据
- 提交后创建 `pending_review` 事件

第一版可以先不做真实文件上传。

原因：

- 当前图片 API 保存的是附件元数据
- 真正文件存储方案还没定
- 先让录入路径在 UI 上跑通，比先接复杂上传更稳

完成标准：

- 页面可以提交文字备注
- 页面可以提交图片证据元数据
- 提交后能看到创建成功结果
- 新事件出现在待确认队列

### 10.2 `/review` 待确认队列页面

目标：

- 展示 `pending_review` 事件
- 展示原始备注和附件元数据
- 支持 edit
- 支持 skip
- 支持 confirm

完成标准：

- 用户可以从页面处理待确认事件
- confirm 后事件变成 `confirmed`
- skip 后事件变成 `skipped`
- edit 后事件保持 `pending_review`

### 10.3 首页入口调整

目标：

- 首页不要只是展示 M1 loop
- 首页要有明确入口：

- 新增录入
- 待确认
- 客户
- 任务

完成标准：

- Chao 打开首页能知道下一步该点哪里
- 不需要记 URL
