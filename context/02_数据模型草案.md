## 1. 文档目的

这份文档不是最终 schema，而是开发前的数据模型草案。

目标只有两个：

1. 先把“哪些对象一定存在”说清楚
2. 把未来扩团队最容易返工的字段先预留

---

## 2. 建模原则

### 2.1 以人作为主对象

第一版按“人”为主，而不是先做完整人/公司双主体。

### 2.2 原始内容和结构化内容分开

原始截图、原始照片、原始文字必须能保留。  
自动提取出来的字段是另外一层数据。

### 2.3 待确认是核心状态

不是所有识别结果都直接进正式档案。  
需要有 review 状态来承接确认过程。

### 2.4 为未来团队协作预留字段

第一版虽然只有 Chao 自己用，但数据里要留 owner / creator / reviewer 这些字段。

---

## 3. 核心对象

### 3.1 users

代表谁在使用系统。

建议字段：

| 字段 | 说明 |
|---|---|
| `id` | 主键 |
| `email` | 登录邮箱 |
| `display_name` | 显示名 |
| `role_type` | 当前角色类型，第一版可简单 |
| `manager_user_id` | 未来上级关系预留 |
| `is_active` | 是否启用 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

说明：

- 第一版真实上只会有一个用户
- 但这个表建议一开始就存在

---

### 3.2 parties

客户主表。第一版以“人”为主。

建议字段：

| 字段 | 说明 |
|---|---|
| `id` | 主键 |
| `display_name` | 客户主名称 |
| `company_name` | 所属公司，可空 |
| `handles_json` | 微信、Telegram、手机号等标识 |
| `referral_source_tag` | 介绍来源标签 |
| `status_label` | 轻量人工状态标签 |
| `tags` | 自定义标签数组 |
| `profile_summary` | 客户概览摘要 |
| `last_contact_at` | 最近一次有效联系时间 |
| `last_contact_summary` | 最近一次沟通摘要 |
| `last_contact_event_id` | 最近一次有效沟通事件，便于进入原始依据 |
| `next_followup_at` | 下一次建议跟进时间 |
| `followup_status` | 当前跟进状态 |
| `owner_user_id` | 当前负责人 |
| `created_by_user_id` | 谁创建的 |
| `reviewed_by_user_id` | 最近一次谁确认的 |
| `updated_by_user_id` | 最近一次谁改的 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

说明：

- 这里的 `followup_status` 比复杂 emoji 状态机更关键
- `company_name` 第一版只是属性，不是独立主对象

---

### 3.3 events

一次录入动作沉淀成一条事件。

建议字段：

| 字段 | 说明 |
|---|---|
| `id` | 主键 |
| `party_id` | 关联客户，可空 |
| `source_channel` | 来源渠道，如 pwa / manual / import |
| `content_type` | 内容类型，如 image / text / card_photo |
| `raw_text` | 原始文字内容，可空 |
| `ai_summary` | 自动摘要，可空 |
| `extracted_fields_json` | 自动提取的字段结果 |
| `review_status` | pending_review / confirmed / skipped |
| `occurred_at` | 实际沟通发生时间，可空 |
| `captured_at` | 用户录入时间 |
| `created_by_user_id` | 谁录入的 |
| `reviewed_by_user_id` | 谁确认的 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

说明：

- `occurred_at` 和 `captured_at` 建议区分
- 用户补录历史记录时，两者可能不同

---

### 3.4 attachments

原始图片附件表。

建议字段：

| 字段 | 说明 |
|---|---|
| `id` | 主键 |
| `event_id` | 归属事件 |
| `storage_key` | 存储路径 |
| `file_name` | 原始文件名 |
| `mime_type` | 文件类型 |
| `file_size` | 文件大小 |
| `width` | 图片宽度，可空 |
| `height` | 图片高度，可空 |
| `created_at` | 创建时间 |

说明：

- 第一版核心是图片，不要求把所有文件类型都覆盖

---

### 3.5 tasks

跟进任务表。

建议字段：

| 字段 | 说明 |
|---|---|
| `id` | 主键 |
| `party_id` | 关联客户 |
| `source_event_id` | 来源事件，可空 |
| `task_type` | 承诺 / 提醒 / 跟进 |
| `description` | 任务描述 |
| `due_at` | 到期时间，可空 |
| `status` | open / done |
| `created_by_user_id` | 谁创建的 |
| `completed_by_user_id` | 谁完成的，可空 |
| `completed_at` | 完成时间，可空 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

## 4. 推荐枚举

### 4.1 review_status

- `pending_review`
- `confirmed`
- `skipped`

### 4.2 followup_status

- `up_to_date`
- `due_soon`
- `overdue`
- `unknown`

### 4.3 task_status

- `open`
- `done`

### 4.4 content_type

- `image`
- `text`
- `card_photo`

### 4.5 source_channel

- `pwa`
- `manual`
- `import`

---

## 5. 第一版建议不单独建表的东西

以下内容第一版不建议为了“以后可能会用”就提前拆复杂：

- 公司实体表
- referral 明细表
- 关系图谱表
- 复杂审计流水表

如果后面明确需要，再拆。

---

## 6. 关键衍生逻辑

### 6.1 录入后更新 party

每次事件从 `pending_review` 进入 `confirmed` 后，系统应尝试回写 `parties`：

- `last_contact_at`
- `last_contact_summary`
- `last_contact_event_id`
- `next_followup_at`
- `followup_status`

客户详情页第一屏可以用 `last_contact_event_id` 找到最近一次沟通对应的截图、附件和原始备注。  
这能让“最新沟通摘要”有原始依据支撑，而不是只显示一段没有出处的文字。

### 6.2 任务和跟进状态互相影响

- 明确承诺或下一步可以生成任务
- 任务是否完成会影响客户的跟进健康度
- 如果没有明确任务，也要允许通过分层策略生成建议跟进时间

---

## 7. 当前结论

第一版数据模型重点不是“大而全”，而是：

- 有清楚的主对象
- 有原始证据
- 有确认状态
- 有任务
- 有跟进状态
- 有未来团队扩展字段
