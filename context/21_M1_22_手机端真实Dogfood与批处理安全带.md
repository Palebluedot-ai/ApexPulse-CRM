# M1.22 手机端真实 Dogfood 与批处理安全带

## 1. 目的

这份文档定义 M1.22 的需求、边界、开发顺序和验收方式。

这份文档是自包含文档。

阅读这份文档不需要依赖其他文档。

M1.22 的核心问题不是“再加一个大功能”。

M1.22 的核心问题是：

```text
当真实微信截图开始从手机进入系统时，系统不能乱、不能误入库、不能让用户迷路。
```

## 2. 当前真实进度

当前已经完成的能力：

- 用户可以登录
- 用户可以在网页上传图片
- 图片会保存到本机项目的 `data/attachments/`
- 图片会创建待确认事件
- 待确认事件可以调用外部视觉 API 提取字段
- AI 提取结果只会回填待确认字段
- 用户可以人工修改客户名、公司、来源、需求、下一步、下次跟进时间
- 用户确认后，事件才会写入客户时间线
- 如果没有绑定现有客户，但填写了客户名，系统可以创建新客户
- 确认入库后可以更新客户最新沟通摘要和跟进状态
- 如果填写了下一步，系统可以生成 follow-up 任务
- 客户详情页可以看到最新沟通卡片
- 任务页可以看到自动生成的任务
- Review 页面默认隐藏明显测试记录
- 测试记录仍然可以切换查看

当前已经跑通过的真实闭环：

```text
真实微信截图 -> 上传 -> 外部视觉 API -> Review 人工修正 -> 确认入库 -> 新客户 -> 自动任务
```

所以，当前不是“完全没有端到端”。

当前已经有电脑浏览器端的真实端到端。

还没有完成的是：

```text
手机端真实端到端 dogfood 的稳定体验。
```

## 3. 手机端端到端的定义

M1.22 里，“手机端端到端”定义为：

1. Chao 在手机浏览器打开本地 CRM。
2. Chao 登录。
3. Chao 从手机相册选择一张真实微信截图。
4. 系统把截图上传到 Mac 本机服务。
5. 系统保存原始图片证据。
6. 系统创建一条待确认事件。
7. Chao 在手机或电脑上进入待确认页面。
8. Chao 点击 AI 提取字段。
9. 系统调用外部视觉 API。
10. AI 结果回填到自然字段。
11. Chao 人工修改不准确字段。
12. Chao 确认入库。
13. 系统创建或更新客户。
14. 系统更新客户最新沟通摘要。
15. 系统生成跟进任务。
16. Chao 可以在手机或电脑上看到客户详情和任务。

只要这 16 步能稳定完成，就可以认为个人版手机端真实 E2E 已经可用。

## 4. 当前离手机端真实 E2E 还差什么

当前主要差三类东西。

第一类是启动和访问方式：

- 本地服务需要能被手机访问
- 手机和 Mac 需要在同一个网络
- 登录 cookie 要能在手机浏览器正常保存
- 手机访问地址需要清楚

第二类是手机上传体验：

- 当前可以选择图片，但一次只处理一张更稳
- 上传成功后需要明确提示下一步
- 多张截图连续上传时，用户需要知道哪些已经进队列
- 图片太大、格式不对、上传失败时要有清楚提示

第三类是 Review 批处理体验：

- 多张截图进入待确认队列时，需要知道当前在处理第几条
- 需要能快速跳到下一条
- 需要避免测试记录、旧记录干扰真实记录
- 需要明确“AI 提取只是回填，不会自动入库”
- 需要防止用户误以为确认了一张图就处理完所有图

## 5. M1.22 需求范围

M1.22 应该做这些事：

- 增加手机端真实 dogfood 启动说明
- 增加手机端访问地址检查
- 增强 Capture 上传后的连续录入提示
- 保持一张图一个待确认事件
- 在 Review 页面强化连续处理状态
- 在 Review 页面显示当前记录范围、当前序号、剩余数量
- 在确认或跳过后，让用户更自然地继续处理下一条
- 增加真实截图 dogfood checklist
- 补充 README 中已经过时的功能状态

M1.22 可以考虑做这些小功能：

- Capture 图片输入支持 `multiple`
- 多选图片时逐张创建待确认事件
- 上传结果显示成功数量和失败数量
- Review 页面增加 `只看有图片证据` 过滤
- Review 页面增加 `AI 未提取 / 已提取` 视觉提示

M1.22 暂时不做这些事：

- 不做自动入库
- 不做批量确认入库
- 不做批量删除
- 不做自动客户合并
- 不做复杂客户去重
- 不做团队权限
- 不做云部署
- 不改成公司服务器优先
- 不要求 PWA 离线可用

## 6. 推荐开发拆解

### 6.1 M1.22A 手机端 Dogfood Runbook

目标：

- Chao 能按文档从手机访问本地服务
- Chao 知道怎么登录
- Chao 知道怎么上传第一张真实截图
- Chao 知道失败时先看哪里

验收：

- 文档写清 Mac 和手机同网
- 文档写清从 dev server 输出里找 Network URL
- 文档写清默认登录账号和密码
- 文档写清手机端第一条真实 case 的验证路径

### 6.2 M1.22B Capture 连续上传体验

目标：

- 手机上传图片后不会迷路
- 上传成功后能继续上传下一张
- 上传成功后能直接去待确认

建议先不做复杂批量上传。

第一版更稳的做法是：

```text
一张图 -> 一个 pending review event -> 上传成功 -> 继续上传或去 Review
```

原因：

- 错误更容易定位
- Review 更容易核对
- 不会一次性制造过多真实记录

验收：

- 上传成功后显示 event id
- 上传成功后显示“继续上传下一张”
- 上传成功后显示“去待确认处理”
- 手机端布局不挤

### 6.3 M1.22C Review 连续处理安全带

目标：

- 多条真实截图在 Review 队列里时，用户知道处理进度
- 用户知道当前过滤条件下还剩几条
- 用户知道 AI 提取不会自动入库
- 用户确认或跳过后，页面自然显示下一条

验收：

- 顶部显示当前范围下的待处理数量
- 每张卡片显示 `第 N 条 / 当前筛选共 M 条`
- AI 提取按钮旁边说明“只回填字段，不会入库”
- 确认或跳过后，当前卡片消失，下一条自然上移
- 空状态文案提示可以切换记录范围

### 6.4 M1.22D README 状态修正

目标：

- 根目录 README 不再说“没有真实上传、没有 Vision API、没有登录”
- README 反映当前真实能力
- README 写清当前仍然不是云部署

验收：

- README 当前阶段写成 `M1 手机端 dogfood 准备中`
- README 明确已有登录、图片上传、Vision API、周报
- README 明确仍未做团队权限、云部署、客户合并、自动入库

## 7. 测试闭环

M1.22 每个业务规则仍然先写测试。

需要覆盖的测试：

- Review 队列过滤不会隐藏真实记录
- Review 队列能正确统计当前可见记录数
- Capture 上传成功后返回 event id
- Capture 拒绝非图片文件
- Capture 空文件或无文件时返回可读错误
- AI 提取失败时 event 仍然保持待确认
- confirm 后仍然只处理当前 event，不影响其他 pending event

需要跑的验证命令：

```bash
pnpm check
pnpm build
```

需要做的手工验证：

```text
手机打开 CRM -> 登录 -> 上传微信截图 -> Review -> AI 提取 -> 人工修正 -> 确认入库 -> 客户详情 -> 任务
```

## 8. 什么时候可以手机真实测试

结论：

```text
现在就可以做第一轮手机端真实测试，但它还是工程 dogfood，不是稳定产品验收。
```

当前手机测试的前提：

- Mac 和手机在同一个 Wi-Fi
- Mac 上 Postgres 正在运行
- Mac 上 Next dev server 正在运行
- `.env.local` 已经有视觉 API 配置
- 手机浏览器能访问 Mac 的 Network URL

第一轮手机测试建议只用 1 张真实微信截图。

不要一上来连续上传 18 张。

原因：

- 当前还没有专门的手机端批处理安全带
- 一次太多真实记录会增加人工清理成本
- 先验证手机路径是否通，再验证批量体验

M1.22 完成后，可以开始第二轮手机测试：

```text
连续上传 3 到 5 张真实微信截图，并逐条 Review。
```

如果第二轮稳定，再考虑：

```text
连续上传 10 到 20 张截图。
```

## 10. 风险和处理方式

如果手机打不开 Network URL：

- 先确认手机和 Mac 是否同一个 Wi-Fi
- 再确认 Mac 防火墙是否拦截
- 再确认 dev server 是否还在运行
- 再确认访问的是 Network URL，不是 localhost

如果手机上传失败：

- 先换一张截图试
- 再确认图片格式是否是 `image/*`
- 再确认终端是否有服务端报错

如果 AI 提取失败：

- 不要重新上传同一张图很多次
- 先保留 pending review event
- 可以手工填写字段后确认入库
- 再单独排查 API 配置或模型返回

如果 AI 识别错客户名：

- 这是预期内风险
- 必须人工修正
- 不允许自动入库

如果生成了不想要的真实客户：

- 先不要删除数据库
- 记录 event id 和 customer id
- 后续单独设计归档、合并或删除策略
