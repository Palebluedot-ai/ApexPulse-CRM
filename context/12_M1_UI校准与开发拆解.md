## 1. 文档目的

这份文档记录 M1 基础闭环跑通后的第一轮 UI / UX 校准。

它不是新的 PRD，也不是新的视觉稿。  
它只回答：

- 当前哪里让 Chao 觉得不对
- V0 的 Web UI 哪些地方值得借鉴
- 下一轮开发应该怎么拆
- 哪些事情先不要做

这份文档自包含，不要求先阅读其他文档。

---

## 2. 当前背景

当前 V1 已经能跑通 M1 第一条真实闭环基础版：

- 本地数据库可以运行
- 文字和图片录入可以保存原始内容
- 待确认队列支持 edit / skip / confirm
- confirm 后可以刷新客户最新沟通字段
- 客户列表和客户详情第一屏可以展示
- 任务可以创建、展示、完成、恢复

但 Chao 亲自打开本地页面后，反馈是：

- 功能看起来能跑通
- 整体视觉和使用感觉仍然有不少怪异
- 客户列表一个客户占用一整行，空间利用率太低
- Chao 不是 UI / UX 专家，需要系统在设计层面提供更多支持

这说明当前阶段不应该继续盲目扩功能。  
下一步应该先做 M1 UI dogfood 和信息密度校准。

## 4. V0 客户列表可借鉴点

V0 客户列表的核心特点：

- 左侧有稳定导航
- 顶部有统计卡片
- 搜索框在列表上方
- 筛选按钮在搜索下方
- 排序控件在右侧
- 客户以 3 到 4 列卡片网格展示
- 每张客户卡片只保留决策字段

客户卡片中最有价值的信息：

- 客户名
- 公司 / 来源 / 阶段
- 下一步动作
- 跟进状态或未跟进天数
- 标签
- 沟通记录数量

这比“一个客户占一整行”更适合高频销售工作台。

原因：

- Chao 未来可能有几百个客户
- 客户列表的第一目标不是阅读长摘要，而是快速扫一遍谁该跟
- 卡片网格能让屏幕一次露出更多客户
- 每张卡片只显示行动线索，点进详情再看完整信息

## 5. 当前 V1 客户列表问题

当前 V1 `/customers` 的主要问题：

- 一名客户占一整行，视觉空间浪费
- 列表更像详情摘要，而不是销售工作台
- 缺少顶部统计，用户不知道整体池子状态
- 缺少搜索和筛选的界面位置
- 跟进状态没有形成强优先级扫描

这些不是数据模型问题，而是信息架构和 UI 密度问题。

---

## 6. 下一轮客户列表调整目标

下一轮应把 `/customers` 改成更接近 V0 的紧凑客户看板。

调整目标：

- 客户从单列长行改成响应式卡片网格
- 桌面端优先支持 3 到 4 列
- 手机端仍保持单列
- 顶部增加统计卡片
- 增加搜索框位置
- 增加筛选按钮位置
- 增加排序控件位置
- 每张卡片优先显示下一步动作和跟进状态

第一版可以先实现静态搜索 / 筛选 UI 位置，也可以直接用 query 参数支持基础过滤。  
推荐优先做 query 参数过滤，因为成本不高，并且能让 dogfood 更真实。

## 7. 开发拆解

### 7.1 文档层

需要更新：

- 当前 UI 校准文档
- 文档总览
- 实施记录

目标：

- 明确客户列表密度问题已被记录
- 明确 V0 PNG 是 UI 参考资产
- 明确下一步先做客户列表 UX 校准

### 7.2 服务层

建议新增或扩展客户 dashboard 服务：

- 计算客户总数
- 计算需要关注客户数
- 计算 `due_soon` 客户数
- 计算 `overdue` 客户数
- 支持基础搜索
- 支持基础状态筛选
- 支持基础排序

这些逻辑要写测试。

### 7.3 页面层

调整 `/customers`：

- 顶部统计区
- 搜索输入
- 筛选按钮
- 排序控件
- 响应式客户卡片网格
- 卡片内突出下一步和状态

暂时不做复杂交互动画。

### 7.4 验证层

完成后需要验证：

- `pnpm check`
- `pnpm build`
- 本地请求 `/customers`
- 页面 HTML 中能看到统计、搜索、筛选、客户卡片

如果浏览器自动化工具可用，再做视觉截图验证。  
如果不可用，先用本地 HTTP 验证页面内容。

---

## 8. 本轮暂不处理

本轮不处理：

- 完整设计系统
- 复杂图表
- 客户详情页大改
- 任务页大改
- 多人权限
- OCR / Vision API
- 真实客户数据导入

原因：

- 当前明确问题在客户列表信息密度
- 先解决最刺眼的 UI / UX 问题
- 避免 M1 dogfood 还没完成就继续扩大范围

---

## 9. 当前决策

当前决策：

`先参考 V0，把客户列表从单列长行改成紧凑卡片看板。`

这个决策不改变产品闭环。  
它只是让已经跑通的 M1 更像一个真正能日常使用的销售工作台。
