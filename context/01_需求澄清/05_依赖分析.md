## 1. 文档目的

这份文档说明第一版业务成立依赖什么，不依赖什么。

---

## 2. 业务依赖

### 2.1 必要依赖

| 依赖 | 作用 | 说明 |
|---|---|---|
| PWA 前端 | 用户正式入口 | 第一版唯一正式入口 |
| 数据库存储 | 保存客户、事件、任务 | 无数据库则无法形成 CRM |
| 图片存储 | 保存截图、名片照片原始内容 | 原始证据必须保留 |
| 图像识别能力 | 读取截图、名片照片 | 可以由能看图的 AI 模型承担 |
| 登录机制 | 保护系统入口 | 第一版虽只有一个人，也建议有登录 |

### 2.2 强烈建议依赖

| 依赖 | 作用 |
|---|---|
| 搜索能力 | 快速找回客户和历史记录 |
| 审计字段 | 为未来团队协作预留 |
| 任务机制 | 把记录转成后续动作 |

---

## 3. 第一版不应强绑定的依赖

| 依赖 | 为什么不应强绑定 |
|---|---|
| Vercel | 未来可能不用 |
| Supabase | 未来可能不用 |
| AWS | 第一版不是必须 |
| 公司服务器 | 当前也不是必须 |
| Discord | 第一版不作为正式主入口 |

---

## 4. 部署依赖原则

### 4.1 当前默认

- 首发默认支持 `Mac mini 本地部署`
- 本地优先

### 4.2 未来可迁移

- 可以迁移到 `Vercel + Supabase`
- 可以迁移到 `AWS`
- 可以迁移到 `公司服务器`

所以在需求层面应遵守：

- 不把产品逻辑写死在某一家平台能力里
- 不让“部署方式”反过来决定“业务流程”

---

## 5. 对 V0 的依赖关系

V0 对 V1 的依赖是“参考依赖”，不是“运行依赖”。

可以借鉴：

- 字段抽取逻辑
- 客户匹配逻辑
- 时间线整理思路
- 任务派生规则

不应继续依赖：

- Discord 主入口
- 单文件 Web 服务
- 历史技术债

---

## 6. 当前结论

第一版真正要保住的是：`本地能跑` 和 `未来可迁`，而不是现在就把云方案一次定死。
