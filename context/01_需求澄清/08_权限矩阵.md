## 1. 文档目的

这份文档说明第一版和未来团队版的权限思路。

第一版不会完整实现权限系统，但必须先定义方向，否则以后很容易返工。

---

## 2. 当前阶段权限原则

- 当前真实用户只有 `Chao`
- 第一版默认 `Chao 拥有全部能力`
- 第一版不做严格 owner 隔离
- 但从数据结构上预留未来团队协作所需字段

---

## 3. 未来默认权限方向

当前已确认的未来方向是：

- `主管看全部`
- `下级优先看自己负责的内容`

---

## 4. 角色矩阵

| 能力 | 个人使用者（当前） | 销售成员（未来） | 主管（未来） | 管理员（未来） |
|---|---|---|---|---|
| 登录系统 | 是 | 是 | 是 | 是 |
| 新建客户记录 | 是 | 是 | 是 | 是 |
| 上传截图/备注/名片 | 是 | 是 | 是 | 是 |
| 查看自己客户 | 是 | 是 | 是 | 是 |
| 查看全部客户 | 是 | 否（默认） | 是 | 是 |
| 修改自己客户 | 是 | 是 | 是 | 是 |
| 修改他人客户 | 是 | 否（默认） | 是 | 是 |
| 查看自己任务 | 是 | 是 | 是 | 是 |
| 查看全部任务 | 是 | 否（默认） | 是 | 是 |
| 查看团队活动流 | 是 | 否（默认） | 是 | 是 |
| 查看审计记录 | 是 | 限制 | 是 | 是 |
| 管理用户与权限 | 是 | 否 | 否/部分 | 是 |

---

## 5. 第一版需要预留的数据字段

建议在核心对象上预留：

- `owner_user_id`
- `created_by_user_id`
- `reviewed_by_user_id`
- `updated_by_user_id`
- `manager_user_id`

---

## 6. 第一版不做的权限能力

- 不做严格行级权限控制
- 不做多团队复杂隔离
- 不做自定义角色编辑器
- 不做上下级审批流

---

## 7. 当前结论

权限这件事第一版的正确做法不是“完全不想”，而是：

- 产品上先简化
- 数据上先留口子
- 未来再逐步收紧
