# Cycle Goal：手机端上线冲刺

## 1. 目的

这份文档定义当前开发周期的 Cycle Goal。

这份文档是自包含文档。

阅读这份文档不需要依赖其他文档。

本周期的核心目标是：

```text
在不砍产品总范围的前提下，加速完成手机端真实使用闭环，并把已有文档沉淀整理成可执行资产。
```

## 2. 周期名称

```text
C1：手机端上线冲刺
```

## 3. 周期时间

建议周期：

```text
2026-05-30 到 2026-06-02
```

这个周期不追求一次性完成长期路线图。

这个周期只追求：

```text
让 Chao 能在手机上真实使用，并且后续开发不会丢上下文。
```

## 4. Cycle Goal

本周期目标：

```text
手机上传真实微信截图 -> AI 提取 -> Review 人工确认 -> 客户入库 -> 任务生成 -> 周报可回顾
```

同时完成：

```text
把当前沉淀文档整理成一套不会漂移的项目执行资产。
```

## 5. 不砍范围的定义

Chao 明确希望不砍范围。

本周期对“不砍范围”的定义是：

- 不删除长期产品目标
- 不否定团队权限、云部署、客户分层、客户合并、AI 辅助等未来能力
- 不把后续功能从路线图里拿掉
- 不为了赶上线把质量门槛拿掉

但本周期也不会把所有长期功能都塞进当前交付。

正确做法是：

```text
总范围保留，周期交付聚焦。
```

也就是：

- 短期做手机端真实可用
- 中期做备份、导出、稳定使用
- 长期做团队、云部署、AI 增强和产品化

## 6. 本周期必须交付

### 6.1 手机端真实 E2E

必须完成：

- 手机可以访问本地 CRM
- 手机可以登录
- 手机可以上传真实微信截图
- 上传后能进入待确认队列
- Review 页面可以 AI 提取字段
- Review 页面可以人工修正字段
- 确认后可以创建或更新客户
- 确认后可以生成 follow-up 任务
- 客户详情页能看到最新沟通
- 任务页能看到下一步

验收方式：

```text
至少 1 张真实微信截图在手机端完整跑通。
```

### 6.2 连续处理安全带

必须完成：

- Capture 上传成功后能继续上传下一张
- Capture 上传成功后能直接去 Review
- Review 页面显示当前筛选范围下的剩余数量
- Review 卡片显示当前是第几条
- Review 明确提示 AI 只回填字段，不会自动入库
- 确认或跳过后，下一条自然上移

验收方式：

```text
连续 3 到 5 张真实截图可以逐条处理，不迷路。
```

### 6.3 文档沉淀

必须完成：

- 当前执行目标更新
- 手机端 dogfood runbook
- 上线 checklist
- README 当前状态修正
- 系统日志追加
- 文档总览更新

验收方式：

```text
下一次新会话可以只读项目文档，就知道当前要做什么。
```

## 7. 本周期可以做，但不强制

这些可以做，但不能影响必须交付：

- 多图选择上传
- 上传成功列表
- AI 提取状态标签
- 手机端视觉 polish
- 周报页面小优化
- 客户详情第一屏小优化

如果时间紧，优先保留主路径。

## 8. 本周期不做，但保留范围

这些不进入本周期开发，但保留在产品总范围：

- Leader / 下级关系
- 团队权限
- 云部署
- 客户合并
- 客户删除
- 客户复杂分层
- 自动生成销售回复草稿
- PWA 离线能力
- 长期监控和运维

这些不是被砍掉。

这些只是没有放进 C1 周期。

## 9. 加速策略

本周期提速方式：

```text
减少等待，增加默认决策；减少大文档，增加可验收小闭环。
```

具体规则：

- Codex 默认推进低风险工程实现
- Codex 默认做 UI 文案和小布局判断
- Codex 默认补测试
- Codex 默认更新系统日志
- 每个小闭环完成后跑验证
- 可以小步 commit
- 不等所有未来需求澄清完才写当前代码

仍然必须暂停确认的事项：

- 删除真实数据
- drop table
- 重做 migration
- 改成自动入库
- 改成团队优先
- 改部署方向为云优先
- 提交真实 API key
- 改权限模型

## 10. 质量门槛

本周期不牺牲以下质量门槛：

- 业务规则先有测试
- `pnpm check` 必须通过
- `pnpm build` 必须通过
- `.env.local` 不进入 Git
- `data/` 不进入 Git
- 真实截图不进入 Git
- AI 结果必须经过 Review
- 原始证据必须保留
- 每轮关键决策写入文档或系统日志

如果赶时间，只能砍非必要功能。

不能砍这些质量门槛。

## 11. 文档整理目标

本周期文档整理不是为了写漂亮文档。

文档整理目标是：

```text
让项目上下文可恢复、可交接、可继续开发。
```

需要形成的文档资产：

- 产品目标：当前产品到底要解决什么
- 当前执行目标：当前周期正在做什么
- 手机 dogfood runbook：Chao 怎么在手机上测
- 上线 checklist：上线前必须确认什么
- 系统日志：每一轮做了什么、为什么做
- 路线图：短中长期功能不丢失

文档写作规则：

- 每份文档必须自包含
- 不依赖交叉引用才能理解
- 关键决策重复写清楚
- 不确定事项写成阻塞或非阻塞
- 不写空泛口号

## 12. 短中长期目标

### 12.1 短期目标

时间：

```text
2026-05-30 到 2026-06-02
```

目标：

```text
手机端真实截图闭环可用。
```

交付：

- M1.22A 手机端 dogfood runbook
- M1.22B Capture 连续上传体验
- M1.22C Review 连续处理安全带
- README 状态修正
- 上线 checklist
- 手机端 1 张真实截图 E2E
- 手机端 3 到 5 张连续截图测试

### 12.2 中期目标

时间：

```text
2026-06-03 到 2026-06-14
```

目标：

```text
每天真实使用，并降低本地运行的数据风险。
```

交付：

- 本地备份策略
- 简单数据导出
- 错误处理优化
- 手机端 UI polish
- Weekly report 可读性优化
- 客户详情第一屏继续优化
- 决定是否迁移到 Vercel / Supabase

### 12.3 长期目标

时间：

```text
2026-06-15 以后
```

目标：

```text
从个人工具逐步走向小团队可用产品。
```

方向：

- 团队权限
- Leader / 下级关系
- 团队 report
- 云部署
- 自动备份
- AI 辅助跟进
- 客户合并和归档
- 更完整的 PWA 安装体验

```text
M1.22A + M1.22B
```

也就是：

## 14. 成功标准

本周期成功标准：

- Chao 可以在手机上完成真实截图录入
- Chao 可以在 Review 中确认 AI 提取结果
- 系统可以创建或更新客户
- 系统可以生成 follow-up 任务
- Chao 不需要记住聊天上下文也能从文档恢复项目进度
- 开发速度提升，但没有牺牲测试、构建、数据安全和 Review-first 原则
