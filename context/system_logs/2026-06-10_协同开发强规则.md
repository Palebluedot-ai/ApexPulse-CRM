# 2026-06-10 协同开发强规则

## 背景

Chao 第一次与另一位协作者通过 GitHub Flow 协同开发。

远端曾出现 18 个新 commit。经检查，这 18 个 commit 已经进入 `origin/main`，不是待合并状态。

本地随后已 rebase 到远端最新 `main`，并把本地文档 commit 推送到远端。

当前结论：

```text
main 和 origin/main 同步。
18 个协作者 commit 已经在 main 上。
不需要再 merge 一次。
后续必须通过 branch + PR 协作。
```

## 本轮新增文件

新增根目录协作规则：

```text
CONTRIBUTING.md
```

该文件自包含说明：

- 禁止不读协作规则。
- PR 是什么。
- main 分支如何保护。
- 如何开分支。
- 如何创建 PR。
- 如何 merge。
- 如何查看 commit。
- 已合并代码出问题时如何 revert。
- 禁止提交哪些 secret 和真实数据。
- 数据库 migration 安全规则。
- 产品边界规则。
- 测试和验证规则。
- AI agent 硬规则。
- GitHub 管理员建议。

## 本轮更新文件

更新：

```text
AGENTS.md
README.md
```

`AGENTS.md` 已增加硬约束：

```text
禁止不读 CONTRIBUTING.md 就修改代码、文档、数据库、部署或 Git 历史。
禁止新工作直接 commit 到 main。
禁止绕过 PR。
禁止不了解当前分支状态就 merge/revert/reset/force-push。
```

`README.md` 顶部已加入协作入口提示。

## 后续建议

建议 Chao 在 GitHub 设置：

- 保护 `main` 分支。
- 要求 PR 才能合并。
- 禁止 force push。
- 要求至少 1 个 reviewer。
- 要求 status checks 通过后才能合并。

如果暂时没有 GitHub Actions，就先人工要求 PR 合并前贴出：

```bash
pnpm check
```

涉及部署、Next.js route、server/client 边界时还要贴出：

```bash
pnpm build
```
