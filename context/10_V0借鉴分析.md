## 1. 目的

这份文档记录 V0 里值得 V1 复现的产品形态和核心逻辑。

V1 不照搬 V0 的技术栈。  
V1 只吸收对产品闭环有价值的东西。

---

## 2. V0 的整体形态

V0 是一个以 Discord 为入口的客户记忆系统。

它已经实现过这些能力：

- 从文字、截图、文件里生成事件
- 用 LLM 抽取客户名、公司、阶段、业务需求、金额、下一步、风险
- 保留原始截图和输入文字
- Web dashboard 展示客户列表
- 客户详情页展示画像、任务、沟通时间线、原始依据
- 根据 extracted fields 生成待办
- 支持 pending / confirmed / skipped 状态

V1 的入口会改成 PWA，但这些产品逻辑仍然有参考价值。

---

## 3. V0 客户详情页的有效结构

V0 客户详情页不是纯档案页。  
它更像一个“客户当前状态页”。

它的主要区域包括：

- 客户标题
- 公司 / 来源 / 介绍人
- 跟进状态标签
- 客户关键事实
- 业务分析
- 待办
- 重要时间点
- 待补充信息
- 沟通时间线
- 原始截图 / 附件 / 录入文字

这说明 V1 的客户详情页不应该只做字段展示。  
它应该优先帮助用户回答：

- 这个人是谁
- 最近发生了什么
- 我下一步该做什么
- 原始依据在哪里

---

## 4. 关于“最新沟通摘要”的结论

最新沟通摘要应该放在客户详情页第一屏。

但它不应该只是一个单行字段。  
更合适的第一版形态是：

`最新沟通卡片`

这张卡片建议包含：

- 最新沟通摘要
- 本次沟通对应的下一步 / 承诺
- 需要补充或确认的信息
- 最新原始截图缩略图
- 查看完整原始依据的入口

原因：

- 销售打开客户页时，最需要先恢复上下文
- 摘要没有原图支撑，用户会不信
- 截图没有摘要，用户要重新读一遍
- 摘要和截图放在一起，最符合真实使用场景

---

## 5. V1 第一屏推荐结构

客户详情页第一屏建议包含：

- 客户名
- 公司名
- 介绍来源标签
- 最近联系时间
- 当前跟进状态
- 下一次建议跟进时间
- 最新沟通卡片

最新沟通卡片内包含：

- 最新沟通摘要
- 下一步 / 承诺
- 风险或待补充信息
- 1 到 3 张原始截图缩略图
- 完整原始依据入口

这比单独放一个 `last_contact_summary` 字段更接近真实使用。

## 6. V0 值得保留的核心逻辑

### 6.1 原始依据永远可回看

V0 的每条 timeline item 都可以展开查看原始截图、附件或录入文字。

V1 应保留这个原则。

### 6.2 截图 OCR 和人工备注都要保留

V0 里人工备注常常比截图 OCR 更重要。

V1 不能只保留 OCR 结果。  
用户手写备注也必须作为原始证据的一部分。

### 6.3 最新时间段优先

V0 已经处理过一个问题：

一张截图里可能有多个相隔很久的聊天片段。  
如果早期聊天里出现“明天/后天”，不能覆盖最新聊天里的真实进展。

V1 应继续保留这个逻辑。

### 6.4 任务要来自真实承诺或下一步

V0 已经区分：

- 明确承诺
- 下一步动作
- 普通跟进建议

V1 的任务生成也应该沿用这个优先级。

### 6.5 疑似重复客户不自动合并

V0 支持通过 party 或 name 临时聚合客户。

V1 第一版也应允许重复客户暂存。  
系统可以提示疑似重复，但不要自动合并。

---

## 7. V0 不建议照搬的部分

以下内容不建议进入 V1 首发核心：

- Discord 作为正式入口
- emoji 自动状态机
- 复杂 referral 表
- 关系图谱
- PDF 作为核心输入
- 多频道录入结构

这些内容不是没有价值，只是不适合放进 V1 的第一条闭环里。

---

## 8. 当前产品判断

“沟通摘要包含很多截图”这个想法是对的。

更准确地说：

V1 不应该把摘要和截图分开看。  
摘要是帮助用户快速恢复上下文，截图是让用户确认上下文可信。

第一版可以先做：

- 最新沟通摘要
- 最新截图缩略图
- 原始依据展开

后面再做：

- 多截图横向浏览
- 截图和摘要逐段对应
- 关键句定位
- 从截图直接跳到对应 timeline item
