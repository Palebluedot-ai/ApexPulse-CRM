## 2. 当前项目状态（2026-06-02 快照）

### 2.1 M1 里程碑定义（来自 20_M1到M5产品路线图 + DESIGN.md + 00_文档总览）
M1 = “个人可用闭环”。

核心闭环：
```
手机/电脑上传截图或文字备注 -> 进入 pending_review -> （可选）AI 提取字段 -> 人工确认/编辑/跳过 -> 系统自动创建或绑定客户 -> 刷新客户 last_contact_* + followup_status + nextFollowupAt -> 自动生成 followup 任务 -> 客户列表/详情可见 -> 周报可见
```

M1 必须做到（已达成）：
- 真实图片上传并保留原始证据（local 或 supabase）
- Review 自然字段（客户名、公司、来源、需求、下一步、下次跟进时间、leadQuality、actionRequired 等）
- 确认后自动建客户 + 任务
- 客户第一屏（Morning Brief + open tasks + 最新沟通卡片 + 证据）
- 任务完成/恢复
- 周报
- 登录 gate
- 本地和云端部署准备

M1 刻意不做（产品边界，严禁擅自越过）：
- 团队权限 / Leader 下级视图（M2 话题）
- 自动入库（永远 review-first）
- OCR 全自动
- 复杂客户分层规则（当前只用 nextFollowupAt + 日期算简单状态）
- 客户合并
- 离线 PWA
- Discord 主入口

- `src/app/api/review/vision-extract/route.ts` 硬编码只支持 `local-images/` + 直接 readFile。

**Step 1: 修云端 Vision 支持（TDD，必做，预计 1-2 小时）**
这是本次 Review 发现的 blocker。

业务规则（必须先写测试证明）：
- 任何 pending_review + 图片 attachment（不管 storageKey 是 local-images/ 还是 supabase-images/），只要 mime 是 image/*，就可以触发 AI 提取。
- 提取时使用统一的 `readImageEvidence(storageKey)` 获取 bytes。
- 提取后仍然只回填字段，event 保持 pending_review。
- 错误提示要区分“不是图片”“不支持的存储 key”“Vision provider 失败”。

实施：
1. 在 `src/server/review/vision-review.test.ts` 或新建 `vision-extract-cloud.test.ts` 里，先加失败测试（用 mock storageKey = "supabase-images/..."）。
2. 修改 `src/app/api/review/vision-extract/route.ts`：
   - 移除 `startsWith("local-images/")` 硬编码。
   - 改用 `import { readImageEvidence } from "@/server/capture/image-storage-provider";`
   - 改成 `const imageBytes = await readImageEvidence(attachment.storageKey);`
   - 更新 canExtract 判断为支持两种 key + image mime。
   - 更新错误消息集合。
3. 可能需要给 vision-extract API 传 options 或让 readImageEvidence 内部用当前 env。
4. 运行 `pnpm test src/app/api/review/vision-extract`（或对应 test 文件）直到通过。
5. 再跑全 `pnpm check`。
6. 更新 `context/30_M1_23D_Vercel部署准备.md` 和本文件里的“已知限制”。

验收：本地用 supabase-images/ 风格的 storageKey mock 能走通提取流程（不需要真实 Supabase key 也能测到“下载失败”或 mock 路径）。
