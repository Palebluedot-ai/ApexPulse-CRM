# M1.10 反馈修正与后续方案

## 1. 目的

这份文档整理 2026-05-29 早上 Chao 对当前页面的反馈，并给出后续产品方案。

这份文档是自包含的。读这份文档不需要先读其他文档。

## 2. 当前阶段判断

当前产品仍处在 M1 dogfood 阶段。

这意味着：

- 现在不是最终产品形态
- 很多入口是为了先跑通闭环
- UI 上不能继续暴露工程字段
- 页面要逐步从“开发可用”变成“普通用户可用”

当前应该继续坚持：

- PWA-first
- local-first
- review-first
- person-first
- single-user first

当前不应该突然加入：

- 真实登录系统
- 复杂权限
- 真实文件上传
- OCR
- Vision API
- 云部署
- 真实客户数据

## 3. Login 的后续方案

未来需要登录验证。

但当前不应马上做复杂登录。

建议分三阶段：

### 3.1 当前阶段

当前阶段不做真实 login。

原因：

- 当前只有 Chao 一个用户
- 本地 dogfood 比账号体系更重要
- 过早做登录会增加开发成本，但不会改善核心销售跟进体验

当前只需要在数据模型里保留 owner / creator / reviewer 字段。

这些字段已经存在。

### 3.2 第二阶段

当产品开始部署到 Vercel、Supabase、公司服务器或 AWS 时，再做基本登录。

建议方案：

- 单用户邮箱登录
- 或 magic link
- 或 Supabase Auth

目标不是权限复杂，而是防止外部访问。

### 3.3 第三阶段

当同事开始使用时，再做团队权限。

第一版团队权限建议：

- owner 看全部
- member 看自己
- manager 看下属

更复杂的权限以后再说。

## 5. Capture 页面反馈

Chao 反馈：

`/capture` 右半部分“图片证据元数据”不自然。

真实产品里用户应该直接上传图片，而不是手动输入 storage key、MIME type、文件大小、宽高。

解释：

这个区域之前是开发阶段占位。

它的目的只是先验证后端可以保存附件元数据。

它不是最终产品形态。

当前修正：

- 改成“上传截图 / 名片照片”
- 用户选择图片文件
- 系统自动读取文件名、类型、大小
- 文件本体暂时不上传
- 仍然保存一条待确认记录
- 页面明确说明这是本地占位上传

未来方案：

- 接真实文件上传
- 文件进入本地存储、Supabase Storage、S3 或公司文件服务
- 后端自动生成 storage key
- OCR 或 Vision API 自动识别
- 用户只确认识别结果，不接触技术元数据

## 6. Review 页面反馈

Chao 反馈：

Review 页面直接出现 JSON 字段，不适合没有编程基础的人。

判断：

这个反馈是对的。

Review 页面应该是业务确认页，不是开发调试页。

当前修正：

- 不再显示 JSON 输入框
- 用户只调整摘要
- 确认入库时只选择客户、摘要和跟进状态
- 技术字段隐藏
- “不绑定客户”改成“新客户 / 暂不匹配现有客户”

后续方案：

Review 页面应该逐步变成：

- 系统识别：姓名、公司、渠道、需求、下一步
- 用户确认或改写自然语言字段
- 不展示 JSON
- 高级调试信息最多放在折叠区域，并默认隐藏

关于客户绑定：

第一版应该支持两种情况：

- 绑定已有客户
- 暂时作为新客户线索保留

未来更好的做法是：

- 如果匹配到已有客户，给出建议匹配
- 如果没有匹配，允许“一键创建新客户”

## 7. Tasks 页面反馈

Chao 反馈：

任务页面右侧任务列表应该放左边。

创建任务可以保留，但不需要占那么大。

当前修正：

- 任务列表放左边
- 创建任务表单放右边
- 创建任务表单高度降低
- 任务列表成为页面主视觉

后续方案：

任务页应该以“我现在要处理什么”为主。

创建任务只是辅助动作。

长期可以考虑：

- 今日任务
- 逾期任务
- 本周任务
- 按客户聚合任务

但当前不需要马上做。

## 8. 后续功能路线

建议后续按阶段推进。

### 8.1 M1.10 当前反馈修正

目标：

- 去掉开发占位感
- 不让用户看到 JSON
- 简化 capture
- 调整 tasks 布局

当前已经执行。

### 8.2 M1.11 Review 自然字段化

目标：

- 把 Review 页面从摘要确认，推进到自然字段确认

字段可以包括：

- 客户名
- 公司
- 来源
- 需求
- 下一步
- 下次跟进时间

不展示 JSON。

### 8.3 M1.12 图片真实上传

目标：

- 真实保存截图或名片照片
- 不再只是 metadata

仍然可以先本地存储，不一定马上上云。

### 8.4 M1.13 OCR / Vision 识别

目标：

- 自动从截图、名片照片提取内容

这一步会需要 API key。

做这一步之前必须暂停确认。

### 8.5 M1.14 登录保护

目标：

- 如果要部署到公网，先加登录保护

如果只在本地 Mac mini 跑，可以暂时不做。

### 8.6 M2 团队协作

目标：

- 给下属使用
- 看他们做了什么
- 管理任务和客户跟进

这一步才进入团队权限和 manager 视角。

## 9. 当前结论

Chao 的反馈说明当前产品已经到了一个关键点：

它不只是要能跑，还要从“开发工具”变成“销售每天真的会用的工具”。

下一步的优先级不是继续堆底层能力。

下一步应该优先减少：

- 工程字段
- 技术文案
- 大而空的占位区
- 非业务用户看不懂的配置

当前 M1.10 的修正方向是正确的。
