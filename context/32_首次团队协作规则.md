# 首次团队协作规则

## 1. 文档目的

这份文档定义 ApexPulse CRM 第一次双人协作开发的最低规则。

这份文档是自包含文档。

阅读这份文档不需要依赖其他文档。

当前协作场景是：

```text
Chao 使用当前电脑和当前 AI agent。
另一位协作者使用另一台电脑和另一个大模型。
两边都可能第一次做 GitHub 协同开发。
```

这份文档的目标不是制定复杂流程。

这份文档的目标是：

```text
避免两个人第一次协作时互相覆盖代码、泄露 key、改错产品边界。
```

## 3. 当前产品主线

当前产品主线是：

```text
PWA-first personal CRM for OTC sales follow-up。
```

当前第一用户是 Chao。

当前核心闭环是：

```text
上传截图或文字 -> 待确认 -> AI 提取 -> 人工 Review -> 客户入库 -> 任务生成 -> 周报回顾
```

当前部署主线是：

```text
Vercel + Supabase Postgres + Supabase Storage + Cloudflare 域名。
```

当前不允许任何协作者擅自改成：

- 公司内部合规系统优先
- 多租户 SaaS 优先
- 团队权限优先
- Discord 优先
- 自动入库优先
- Mac mini 自托管优先
- 公司服务器优先

如果确实认为需要改产品边界，必须先问 Chao。

## 4. 两台电脑第一次协作的最低 Git 规则

### 4.1 开始工作前

每次开始工作前，先运行：

```bash
git status -sb
git pull --ff-only
```

如果 `git pull --ff-only` 失败，不要强行 merge。

失败时先停下来，把输出贴给当前负责协调的人。

不要运行：

```bash
git reset --hard
git clean -fdx
git push --force
```

除非 Chao 明确确认。

### 4.2 分支规则

第一阶段可以继续用 `main` 小步推进，但必须满足：

- 每次改动小
- 每次 commit 信息清楚
- push 前确认没有 `.env.local`、`data/`、真实截图或真实 key
- 两个人不要同时改同一批文件

如果两个人要并行开发，建议开分支：

```bash
git checkout -b codex/short-task-name
```

分支命名建议：

```text
codex/m1-23e-vision-supabase
codex/m1-23e-vercel-docs
codex/m1-24-review-polish
```

### 4.3 提交规则

commit message 使用简短英文前缀：

```text
feat: ...
fix: ...
docs: ...
test: ...
chore: ...
```

例子：

```text
fix: support supabase image vision extraction
docs: add first team collaboration rules
```

一个 commit 只做一件事。

不要把 unrelated 改动混在一起。

## 5. 文件所有权规则

第一次协作时，最容易冲突的是同一个文件被两边同时改。

因此先用简单规则：

```text
同一时间，一个任务只能有一个人负责同一组文件。
```

如果一个人正在改：

```text
src/app/api/review/vision-extract/route.ts
src/server/ai/
src/server/review/
```

另一个人就不要同时改这些文件。

如果一个人正在改：

```text
context/31_M1_23E_首次云端部署与C1验收路径.md
context/00_文档总览.md
AGENTS.md
```

另一个人不要同时改这些文件。

需要交叉修改时，先在聊天里说清楚：

```text
我准备改哪些文件，为什么改，预计什么时候结束。
```

## 6. 环境变量和 secret 规则

绝对不能提交：

- `.env`
- `.env.local`
- `.env.*.local`
- Supabase service role key
- Supabase anon key
- Vercel token
- Vision API key
- Postgres connection string
- 真实客户截图
- `data/attachments/`

真实 key 只能放在：

```text
本机 .env.local
Vercel Dashboard Environment Variables
Supabase Dashboard
```

不要把真实 key 发到聊天里。

如果需要确认 key 是否配置成功，只能运行检查命令：

```bash
pnpm env:check staging
```

检查命令只能看 key 是否存在，不打印 key 的值。

## 7. 数据库安全规则

任何人都不能擅自运行：

```sql
drop table
truncate table
delete from
```

任何人都不能擅自：

- 重做 migration
- 删除 migration 文件
- reset Supabase database
- 删除 Supabase bucket
- 删除真实客户数据

安全检查命令：

```bash
pnpm db:check
```

这个命令只读表结构。

运行 migration 前必须确认：

```text
DATABASE_URL 或 MIGRATION_DATABASE_URL 指向的是正确数据库。
```

## 8. 代码质量规则

业务规则必须先写测试。

业务规则包括：

- Review 状态变化
- AI 提取结果归一化
- 图片上传和读取
- 任务生成
- follow-up 状态刷新
- 登录和权限判断
- 数据库迁移检查

完成代码改动后至少运行：

```bash
pnpm check
```

涉及部署、Next.js route、server/client 边界时，还要运行：

```bash
pnpm build
```

如果 `pnpm build` 在 sandbox 里因为 Turbopack 绑定端口失败，可以在允许的本机环境重跑。

如果 build 报真实代码错误，必须修。

## 9. 文档规则

所有产品和协作文档必须自包含。

不要写：

```text
详见某某文档
先读某某文档
参考前文
```

如果一个关键背景对当前文档重要，就在当前文档里重复写清楚。

新增重要文档时，也要更新：

```text
context/00_文档总览.md
```

重要开发节点结束时，要写系统日志：

```text
context/system_logs/
```

## 10. 另一个大模型进入项目时必须遵守

如果另一位协作者使用另一个大模型，必须先把以下规则告诉它：

```text
你正在协作 ApexPulse CRM。
这是 PWA-first personal CRM，不是团队权限优先系统。
不要改产品边界。
不要提交任何 secret。
不要删除数据。
业务规则先写测试。
完成后运行 pnpm check，涉及部署或 Next route 时运行 pnpm build。
文档必须自包含。
```

另一个大模型开始前，应先读取：

```text
AGENTS.md
context/00_文档总览.md
context/31_M1_23E_首次云端部署与C1验收路径.md
context/32_首次团队协作规则.md
```

这些文件不是为了制造阅读负担。

这些文件是为了避免它从零开始猜项目方向。

## 11. 第一次协作推荐分工

推荐先不要两个人同时写核心代码。

第一次协作更稳的分工是：

### Chao / 当前 Codex

负责：

- M1.23E 主路径
- Supabase / Vercel 部署验证
- vision-extract 支持 `supabase-images/`
- C1 手机真实 E2E

### 新协作者 / 另一台电脑

负责：

- 阅读项目文档
- 本地跑起来
- 检查 README / onboarding 是否能看懂
- 找不清楚的地方
- 不直接改核心业务逻辑

第一轮协作目标不是并行写很多代码。

第一轮协作目标是：

```text
让第二个人能在另一台电脑上成功理解项目、跑起项目、发现文档缺口。
```

这比急着并行开发更重要。

## 12. 冲突处理

如果出现 git conflict：

1. 不要 panic。
2. 不要 reset。
3. 先运行 `git status -sb`。
4. 看冲突文件。
5. 判断是谁负责这个文件。
6. 如果不确定，停下来问。

冲突处理原则：

```text
保留用户真实改动，保留已验证逻辑，不为了快速消除冲突乱删代码。
```

```text

```

push 后，另一台电脑可以：
