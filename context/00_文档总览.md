这份文档是整个 `context/` 目录的总览页。

目标：

1. 让未来任何一次新会话都能快速恢复上下文
2. 让文档体系尽量自包含
3. 明确当前哪些已经确认，哪些还没定死

---

## 2. 当前确认度结论

### 2.1 已经足够确认，可以进入开发设计

下面这些已经可以视为当前阶段的“已确认事实”：

- 项目定位：`side project`
- 当前唯一真实用户：`Chao`
- 第一版正式入口：`PWA`
- 第一版支持输入：`截图 / 文字备注 / 名片照片`
- 第一版以“人”为主组织客户档案
- 原始证据必须保留
- 自动识别结果先进入待确认队列
- 每次新增正式记录后，系统要自动刷新跟进状态
- 第一版需要任务体系
- 第一版部署默认支持 `Mac mini 本地优先`
- 第一版不做复杂权限，但要预留 owner / creator / reviewer 等字段

### 2.2 已确认原则，但未定死规则

下面这些已经定了方向，但还没定细则：

- 分层提醒要做，但分层规则和默认天数未定
- 未来主管看全部、下级看自己，但权限细节未定
- 回复起草可能做，但不属于首发必需闭环
- 公司未来可能升级为独立主对象，但首发先不做
- Discord 未来可能作为可选导入口，但不是当前正式入口
- 最终部署未来可能迁云，但首发不以云方案为前提

### 2.3 当前没有发现会阻塞开发的关键不确定项

也就是说：

- 不是“100% 所有细节都确认了”
- 但已经足够支持进入 `PRD -> 开发拆解 -> 真正实施`

## 3. 文档是否自包含

### 3.1 结论

当前文档体系已经 `基本自包含`，足够让下一个 agent 或下一个 milestone 接着做。

### 3.2 当前关于“自包含”的要求

从现在开始，这套文档遵守下面这个原则：

- 每份文档都应能单独成立
- 不依赖“先读另一份文档”才能理解
- 不使用“详见某文档”“阅读顺序如下”这类链式结构
- 如果某个关键背景对当前文档成立是必要的，就直接在当前文档里重复写清楚

### 3.3 这份总览页的作用

这份文档只负责说明：

- 当前确认度如何
- 文档体系是否足够进入开发
- 下一次 milestone 是什么

它不是其他文档的前置依赖。

## 4. BMAD / Document Driven Development 下的下一次 milestone

当前文档阶段结束后，下一次 milestone 不应该直接叫“开始写代码”，而应该是：

`M1：打通首个可演示的录入-确认-入库-跟进刷新闭环`

这个 milestone 的目标不是做完整产品，而是完成最小垂直切片：

1. 在 PWA 上传截图或输入备注
2. 系统保存原始内容
3. 系统生成待确认结果
4. 用户确认后生成客户事件
5. 客户的最新跟进状态被刷新
6. 如有需要，自动生成任务
