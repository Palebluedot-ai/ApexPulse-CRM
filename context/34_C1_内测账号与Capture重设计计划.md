# C1 内测账号与 Capture 重设计计划

## 1. 目的

这份文档是自包含文档。

阅读它不需要依赖其他文档。

本文件用于固定 C1 下一阶段的开发边界和 to-do list，避免在内测账号、Capture 页面和 UI 原型之间反复来回改，最后失去主线。

当前产品已经可以通过 `https://www.palebluedot.pro` 进入线上环境。下一阶段不是进入完整团队版，而是支持少量内测用户参与测试，并把手机端 `/capture` 页面改得更像真实产品入口。

## 2. 当前背景

Chao 计划下周让不到 10 个新用户加入内测。

这些用户的目的主要是测试产品，不是正式长期使用。

这些用户不能共用 Chao 当前的同一个登录账号和密码。

同时，当前 `/capture` 页面仍然有明显原型痕迹：

- 页面分左右两栏，文字输入和图片输入并列，手机端不够聚焦。
- 图片上传区露出浏览器原生 `Choose File / No file chosen`，字体和产品视觉不一致。
- “保存图片证据”按钮在未选择图片时显得唐突。
- 旧文案仍提到“本机 data/attachments/”和“先不接 OCR”，这已经不符合当前线上 Supabase + Vision 状态。
- Chao 希望上传页尽量简洁，最好是一个明显的加号，用户自然知道可以上传图片。

## 3. 产品边界

本阶段仍然是 C1 内测，不是 M2 团队权限系统。

本阶段允许：

- 支持多个内测账号登录。
- 每个内测用户用自己的邮箱登录。
- 每个用户的录入、待确认、客户、任务默认归属到自己。
- Chao 可以继续作为 owner 使用现有功能。
- Capture 页面做大幅简化和交互重设计。
- 做临时 `/capture-lab` 页面帮助 Chao 快速确认设计方案。

本阶段不做：

- 自助注册。
- 忘记密码。
- 邮箱验证码。
- 完整团队管理后台。
- Leader 查看下级全部数据。
- 平级数据权限矩阵。
- 客户转移。
- 审计日志。
- 自动跳过 Review。

如果要做 Leader / 下级关系，必须进入 M2，并重新写独立权限计划。

## 4. 内测账号方案

### 4.1 推荐方案

使用 invite-only 多账号。

每个内测用户使用真实公司邮箱作为账号。

初始密码由 Chao 或系统管理员生成并单独发给内测用户。

用户第一次登录后暂时不强制改密码。因为这是 C1 内测，不是长期正式上线。

### 4.2 为什么不用共用测试账号

不用共用测试账号，原因是：

- 无法区分是谁上传了截图。
- 无法判断谁触发了哪个 bug。
- 客户和任务会混在一起。
- 后续从测试进入真实使用时迁移困难。
- 用户行为反馈没有归属。

### 4.3 为什么暂不做自助注册

暂不做自助注册，原因是：

- 用户不到 10 个，手动创建更快。
- 自助注册会引入邮箱验证、密码重置、垃圾注册、风控等额外问题。
- 当前目标是验证 CRM 主闭环，不是验证注册系统。

### 4.4 技术方向

当前登录逻辑是单账号环境变量：

```text
LOCAL_AUTH_EMAIL + LOCAL_AUTH_PASSWORD
```

这不能满足内测多账号。

建议改成：

```text
users.email + users.password_hash + users.is_active
```

登录时：

1. 用户输入 email 和 password。
2. 后端用 email 查 `users` 表。
3. 如果用户不存在或 `is_active=false`，返回登录失败。
4. 如果密码 hash 不匹配，返回登录失败。
5. 如果通过，创建现有 session cookie。

### 4.5 密码存储

密码不能明文存数据库。

建议使用 Node 内置 `crypto.scrypt` 或 `crypto.pbkdf2` 实现密码 hash。

第一版可以不用额外安装依赖。

密码 hash 字段建议格式：

```text
scrypt$参数$salt$hash
```

如果后续接入 Supabase Auth、Clerk、Auth.js，再做迁移。

### 4.6 创建内测用户方式

第一版不做 UI 管理后台。

提供一个本地脚本：

```bash
pnpm user:create
```

输入或传入：

```text
email
displayName
temporaryPassword
roleType
```

脚本行为：

- 创建用户。
- 写入 password hash。
- 如果用户已存在，默认不覆盖密码，除非显式传 `--reset-password`。
- 不打印真实密码。
- 不把任何 secret 写入 Git。

## 5. Capture 页面目标

### 5.1 核心目标

`/capture` 应该让用户第一眼知道：

```text
点加号 -> 上传截图 -> 进入待确认 -> AI 提取 -> 人确认后入库
```

页面不应该像后台表单。

页面应该像手机端产品入口。

### 5.2 推荐信息架构

首屏只保留一个主动作：

```text
+ 上传截图
```

图片选择后再显示：

- 图片预览。
- 可选备注输入框。
- 上传按钮。
- 上传中状态。
- 上传成功后的去待确认按钮。

文字备注作为次入口：

```text
没有图片？写一条文字备注
```

默认折叠。点击后展开。

### 5.3 必须删除或替换的旧文案

删除：

```text
图片会保存到本机 data/attachments/
```

删除：

```text
先不接 OCR
```

替换为：

```text
上传后进入待确认，AI 会先提取字段，你确认后才入库。
```

或者更短：

```text
上传截图，AI 先提取，你确认后入库。
```

### 5.4 文件选择控件要求

不要直接显示浏览器原生：

```text
Choose File
No file chosen
```

应该隐藏原生 input，用自定义按钮触发：

```text
<input type="file" hidden />
```

自定义上传区可以是：

```text
圆形加号 + 上传截图
```

选择图片后显示文件名和预览。

### 5.5 上传按钮出现规则

未选择图片前，不显示“保存图片证据”或“上传到待确认”主按钮。

选择图片后，显示：

```text
上传到待确认
```

上传中显示：

```text
上传中…
```

上传成功后显示：

```text
已进入待确认，AI 正在提取，约 20 秒。
```

并提供：

```text
去待确认
继续上传
```

## 6. 临时设计确认工具

### 6.1 目标

为了减少 back and forth，先做一个临时页面：

```text
/capture-lab
```

这个页面用于让 Chao 快速比较 2 到 3 个 Capture 方案。

这个页面不接真实 API，不写数据库，不上传文件。

它只用于确认交互和视觉方向。

### 6.2 方案候选

方案 A：极简加号卡片。

- 页面中央一个大加号。
- 一句短文案。
- 点击后进入预览状态。
- 最接近 Chao 描述的“看到加号自然知道上传”。

方案 B：相机胶片卡片。

- 上传区更像图片收集盒。
- 适合连续上传多张截图。
- 可以显示最近上传队列。

方案 C：聊天证据入口。

- 更强调“微信截图 / 名片 / 备注都是证据”。
- 适合后续扩展多渠道，但第一屏可能信息稍多。

### 6.3 使用方式

开发时先实现 `/capture-lab`。

Chao 在浏览器或手机上看这 2 到 3 个方案。

Chao 选一个方向后，再把选中的方案落到正式 `/capture`。

落地后删除 `/capture-lab`，不把临时页面长期留在线上。

## 8. 推荐 PR 拆分

不要把所有事情塞进一个 PR。

推荐拆分：

```text
PR 1: docs(c1): beta account and capture redesign plan
PR 2: feat(capture): add temporary capture-lab prototypes
PR 3: feat(capture): simplify production capture upload flow
PR 4: feat(auth): invite-only beta accounts
```

如果时间很赶，可以合并 PR 2 和 PR 3，但不建议把 PR 4 也混进去。

原因：

- Capture 页面是 UI 体验问题。
- 多账号是登录和数据归属问题。
- 两者风险类型不同。

立即下一步：

