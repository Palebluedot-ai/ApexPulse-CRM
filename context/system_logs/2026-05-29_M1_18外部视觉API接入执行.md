# 2026-05-29 M1.18 外部视觉 API 接入执行

## 本轮目标

本轮目标是接入外部视觉 API 的最小闭环。

这一步只做 AI 辅助提取。

这一步不做自动入库。

AI 提取结果必须回到 Review 页面，由用户人工确认后才会写入客户、任务和跟进状态。

## 已确认的产品事实

Chao 已确认：

- API provider 已经有
- API key 已经有
- 未来真实截图允许发送给外部 API

本轮不把真实 key 写入代码。

本轮不提交 `.env.local`。

本轮代码只读取环境变量。

## 本轮新增环境变量

需要在本机 `.env.local` 填：

```text
VISION_API_KEY=你的真实 key
VISION_API_BASE_URL=https://api.openai.com/v1
VISION_API_MODEL=你的视觉模型名
```

第一版 adapter 使用 OpenAI-compatible chat completions 请求格式。

如果最终 provider 不是 OpenAI-compatible，只需要替换 provider adapter，不改 Review 主流程。

## 本轮已完成内容

已新增视觉提取标准化模块：

```text
src/server/ai/vision-extraction.ts
```

能力：

- 从 provider 返回中提取文本
- 支持 OpenAI-compatible chat completion 返回格式
- 支持 OpenAI responses-style 输出格式的文本提取
- 支持解析纯 JSON
- 支持解析 markdown code fence 包裹的 JSON
- 将结果标准化为 Review 自然字段
- 生成带 `aiExtractionSource=vision_api` 的 extracted fields

已新增视觉 provider adapter：

```text
src/server/ai/vision-provider.ts
```

能力：

- 从 env 读取 `VISION_API_KEY`
- 从 env 读取 `VISION_API_BASE_URL`
- 从 env 读取 `VISION_API_MODEL`
- 调用 `${VISION_API_BASE_URL}/chat/completions`
- 以 data URL 方式发送本地图片
- provider 失败时抛错，不返回半截字段

已新增 Review patch helper：

```text
src/server/review/vision-review.ts
```

能力：

- 将 AI 提取结果合并到 existing extracted fields
- 保留原有未冲突字段
- AI 摘要为空时回退到当前摘要或原始备注
- 不改变 event 的 review status

已新增 Review API：

```text
POST /api/review/vision-extract
```

行为：

- 必须登录
- 只处理 `pending_review` event
- 只处理本地图片附件
- 读取本地图片文件
- 调用外部视觉 API
- 回填 summary 和 extracted fields
- event 仍保持 `pending_review`
- 返回自然字段给前端

已更新 Review 页面：

- 有本地图片附件时显示 `AI 提取字段`
- 点击后调用 `/api/review/vision-extract`
- AI 结果会回填到当前表单
- 页面提示用户检查后再确认入库

## 真实 API Dogfood 结果

Chao 已在本机 `.env.local` 填入真实视觉 API 配置。

本次验证没有记录真实 API key。

本次验证没有使用真实客户截图。

本次验证使用本地生成的 demo 图片。

验证过程：

- 1x1 demo PNG 调用失败，provider 返回图片宽高过小，要求宽和高至少 8 像素
- 16x16 demo PNG 调用失败，provider 返回总像素 256 低于最低 512 像素
- 32x32 demo PNG 调用成功

成功调用结果：

```text
HTTP 200
reviewStatus=pending_review
extractedFields.aiExtractionSource=vision_api
naturalFields 为空字段
```

因为 32x32 demo 图片没有真实文字、客户、公司或需求，返回空自然字段是合理结果。

本次验证结论：

- `.env.local` 配置可以被本地服务读取
- 外部视觉 API 链路可以打通
- 当前 adapter 的 OpenAI-compatible chat completions 格式可用
- AI 提取结果只回填 Review 字段
- event 没有被自动确认
- 没有自动创建客户
- 没有自动生成任务

下一步建议：

使用 1 张低风险真实截图或名片照片做第二轮 dogfood。

第二轮 dogfood 的判断标准：

- 能否提取客户名
- 能否提取公司

- 是否会胡编图片里没有的信息
- 回填后人工确认入库是否顺手

## M1.18.2 提取质量校准

Chao 已确认 `data/IMG_3877.PNG` 到 `data/IMG_3894.PNG` 共 18 张截图可以发送到外部视觉 API 做测试。

本次测试结果：

- 18 张截图全部 API 返回 `200`
- 模型能大体识别对方昵称或姓名
- 模型能生成可用的中文 CRM 摘要
- 开户、资料交换、约电话、约咖啡、约见面等场景能被识别为需要跟进
- 普通自我介绍场景大多被识别为 `unknown`

发现的问题：

- 有时会把群名、HashKey 或我方公司误当成对方公司
- 有时会把普通寒暄标成 `warm`
- 有时会把 `周一`、`下周` 这类相对时间硬推成具体日期
- 有时会把 `confidence` 输出成数字，而不是 `high / medium / low`

已完成校准：

- prompt 明确要求 `companyName` 只有明确属于对方时才填写
- prompt 明确禁止把 HashKey、群名、我方公司误当对方公司
- prompt 明确要求 `nextFollowupAt` 只有明确日期才填写
- prompt 明确要求 `confidence` 只能是 `high / medium / low`
- parser 会丢弃 `下周一` 这类模糊跟进日期
- parser 支持 `counterpartyName` 作为 `customerName` 的别名
- extracted fields 增加 `phone`
- extracted fields 增加 `email`
- extracted fields 增加 `telegram`
- extracted fields 增加 `wechatAlias`
- extracted fields 增加 `leadQuality`
- extracted fields 增加 `confidence`
- extracted fields 增加 `actionRequired`
- extracted fields 增加 `evidenceNotes`

已新增本地测试脚本：

```text
scripts/vision-dogfood.ts
```

用途：

- 批量调用当前 `.env.local` 配置的视觉 API
- 复用生产 prompt
- 复用生产 parser
- 输出标准化后的 `result`
- 输出产品实际会保存的 `extractedFields`
- 不打印 API key

回归测试结果：
