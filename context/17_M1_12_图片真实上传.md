# M1.12 图片真实上传

## 1. 文档目的

这份文档记录 M1.12 的产品目标、实现边界和验证结果。

这份文档是自包含的。读这份文档不需要先读其他文档。

## 2. 当前问题

M1.10 后，Capture 页面已经从“手填图片元数据”改成“选择图片”。

但当时仍然只是占位上传。

系统只保存文件名、类型、大小和备注。

图片文件本体没有真正保存。

这会导致一个核心问题：

CRM 里有证据记录，但无法回看原始图片证据。

对于截图驱动的 sales follow-up，这不够。

## 3. 本轮目标

M1.12 的目标是让图片证据真的保存到本机。

用户在 `/capture` 选择截图或名片照片后：

- 后端接收真实文件
- 文件写入本地目录
- 数据库保存附件记录
- 事件进入待确认队列
- Review 页面可以打开本地图片证据

## 4. 本轮产品边界

本轮坚持下面边界：

- 不上云
- 不接 Supabase Storage
- 不接 S3
- 不接公司文件服务
- 不接 OCR
- 不接 Vision API
- 不改数据库表结构
- 不做 migration
- 不处理真实客户数据

原因：

当前最重要的是先跑通本地证据闭环。

云存储、OCR 和自动识别都可以以后再加。

如果现在一起做，会把存储、识别、成本、错误处理混在一起。

## 5. 本轮实现

图片文件会保存到：

```text
data/attachments/
```

这个目录已经被 `.gitignore` 忽略。

也就是说：

- 本地可以保存图片
- 图片不会进入 Git
- 不会误提交真实截图

系统会生成安全的本地 storage key。

格式类似：

```text
local-images/2026/05/29/20260529T061000000Z-abc123-demo.png
```

系统会清洗文件名。

系统会拒绝：

- 空文件名
- 非图片 MIME type
- 非正数文件大小
- 不安全的本地路径

## 6. Review 页面变化

Review 页附件区现在会判断附件是否是本地保存的图片。

如果是本地图片，会显示：

```text
打开图片证据
```

点击后会通过本地附件 API 打开图片。

旧的占位附件不会强行显示可打开链接。

这样可以避免把历史占位记录误认为真实文件。

## 7. 技术说明

图片上传接口从 JSON metadata 改成 multipart form upload。

前端提交：

- `imageFile`
- `note`

后端处理：

- 读取 `FormData`
- 验证 `imageFile`
- 生成本地 storage key
- 写入 `data/attachments/`
- 创建 pending review event
- 创建 attachment 记录

新增附件读取接口：

```text
GET /api/attachments/:id
```

这个接口只读取安全的本地 storage key。

如果 attachment 不是本地文件，返回 404。

## 8. 当前刻意不做

本轮不做图片缩略图。

原因：

先有“打开证据”的能力即可。

缩略图会涉及更多 UI 密度和性能判断。

本轮不做 OCR。

原因：

OCR 需要 API key、成本和识别准确率评估。

本轮不自动从名片创建客户。

原因：

自动创建客户涉及去重、合并和 owner 规则，需要单独设计。
