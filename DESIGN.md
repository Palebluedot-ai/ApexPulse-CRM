## TL;DR

V1 是一个 `PWA-first` 的个人销售 CRM。  
第一版只服务 Chao 自己，目标不是做“大而全 CRM”，而是把这条闭环跑通：

1. 用截图、文字备注、名片照片快速记录客户互动
2. 把记录整理成按“人”为主的客户档案和时间线
3. 每次新增正式记录后，自动刷新客户的最新跟进状态
4. 自动或半自动生成待跟进任务，避免销售线索沉没

V0 只作为逻辑参考，不作为必须延续的架构。  
部署上首发优先支持 `Mac mini 本地部署`，但架构不排斥未来迁到 `Vercel / Supabase / AWS / 公司服务器`。

---

## 1. 项目定位

### 1.1 当前定位

- 项目性质：`side project`
- 当前主要业务场景：HashKey OTC 销售工作流
- 当前唯一真实用户：`Chao`
- 未来目标用户：Chao 的下级销售同事

### 1.2 这不是一个什么项目

第一版不是：

- 面向外部客户销售的 SaaS CRM
- 标准化企业内部 CRM 平台
- 复杂权限和多租户系统
- 返佣结算系统

第一版是：

- 一个以客户互动记录和跟进提醒为核心的个人 CRM
- 一个先解决“记下来、找得到、跟得上”的销售外接大脑

---

## 2. 为什么要重写

### 2.1 V0 的真正价值

V0 已经证明了一些核心逻辑有价值：

- 截图和备注可以变成客户记录
- 自动提取字段有帮助
- 时间线和任务派生有帮助
- 搜索和找回历史沟通有帮助

### 2.2 V0 的核心问题

V0 更像“工作流原型”，而不是一个可以长期演进的产品形态：

- 入口强依赖 Discord
- 体验割裂
- Web 部分技术债很重
- 很多假设是为了让原型跑起来，不适合继续背进 V1

### 2.3 重写的目标

重写不是为了炫技，而是为了做三件事：

1. 把入口统一到 PWA
2. 把产品边界重新梳理清楚
3. 让 Chao 真的能自己逐步掌控和维护这套系统

## 3. 第一版要解决的核心问题

### 3.1 记录问题

客户互动很多，但没有被及时记录下来，或者记录方式太散。

### 3.2 找回问题

记录即使存在，也很难按客户、时间、上下文快速找回。

### 3.3 跟进问题

最关键的问题不是“有没有存档”，而是：

- 有没有下一步
- 什么时候该跟
- 有没有超期
- 哪些 lead 正在沉没

### 3.4 未来团队问题

未来如果扩到下级成员，主管需要知道：

- 每个人今天做了什么
- 谁的跟进在堆积
- 谁长期没有推进客户
- 谁改过哪些客户信息

## 4. 第一版产品目标

V1 首发只证明下面 6 件事：

1. 可以顺手记录客户互动
2. 可以把记录沉淀成按人查看的客户档案
3. 可以把互动整理成时间线
4. 每次新增正式沟通后，客户的最新跟进状态会自动刷新
5. 可以生成并管理待跟进任务
6. 可以在本地先跑起来，未来再迁移

## 6. 核心产品决策

### 6.1 PWA 优先

第一版只做 PWA，不做原生 App。  
先把录入、查看、跟进闭环跑通，比追求原生体验更重要。

### 6.2 输入内容范围

第一版支持：

- 截图
- 手动文字备注
- 名片照片

第一版不把 PDF / 其他文件当核心输入。

### 6.3 以人为主

客户档案第一版按“人”为主组织。  
公司先作为属性，不先做完整独立主体。

### 6.4 原始证据必须保留

截图、名片照片、原始备注都必须保留。  
系统不能只保留 AI 解释后的结果。

### 6.5 AI 先辅助，用户再确认

第一版默认策略：

- 系统可自动识别客户名、金额、下一步、标签等信息
- 识别结果先进入待确认队列
- 用户确认后再进入正式记录

### 6.6 跟进状态比复杂标签更重要

第一版不先做复杂 emoji 状态机。  
更重要的是每次新增正式记录后，系统要刷新：

- `last_contact_at`
- `last_contact_summary`
- `next_followup_at`
- `followup_status`

### 6.7 默认提醒按分层设计

如果沟通里没有明确下一次跟进时间，系统应按“分层提醒”设计：

- 新线索和高热度客户，提醒更快
- 普通客户，提醒节奏居中
- 沉睡客户，提醒周期更长

当前只确认原则，不确认具体阈值和天数。

### 6.8 Referral 先做轻量

第一版只记录 `介绍来源标签`，不做复杂返佣字段体系。

### 6.9 权限先留骨架，不急着做满

第一版只有 Chao 自己用，不做复杂权限控制。  
但数据结构中要预留：

- `owner_user_id`
- `created_by_user_id`
- `reviewed_by_user_id`
- `updated_by_user_id`
- `manager_user_id`

未来默认方向：

- 主管看全部
- 下级优先看自己

## 7. V0 复用策略

### 7.1 可以借鉴的部分

- 字段提取思路
- 客户匹配逻辑
- 任务派生逻辑
- 时间线组织方式
- 原始证据保留原则

### 7.2 不应照搬的部分

- Discord 主入口
- 单文件 Web 架构
- 历史 raw SQL 方式
- 已确认无价值的死表/死列
- 为旧工作流临时打补丁形成的复杂实现

### 7.3 对 V0 的当前结论

V0 是参考，不是包袱。  
V1 不以“功能完全对齐 V0”作为首发目标，而以“产品闭环成立”作为首发目标。

## 8. 技术方向

### 8.1 选型原则

- 选成熟方案
- 选未来容易维护的方案
- 产品逻辑不绑死单一部署平台
- 能先本地跑，再迁移

### 8.2 当前建议技术栈

| 层 | 技术 |
|---|---|
| 前端 | Next.js 15 |
| PWA | Serwist |
| 样式 | Tailwind CSS |
| 组件 | shadcn/ui |
| 后端 | Next.js Route Handlers / Server Actions |
| ORM | Drizzle |
| 数据库 | PostgreSQL |
| 包管理 | pnpm |
| Node | 22 |

### 8.3 关于部署

当前不把部署方案当真理源，而把它看成可替换层。

首发默认目标：

- 能在 Mac mini 上稳定部署

后续可迁移目标：

- Vercel + Supabase
- AWS
- 公司服务器

### 8.4 关于数据库和对象存储

第一版文档层面只要求：

- 需要关系型数据库
- 需要原始图片存储

至于是本地磁盘、本地对象存储、Supabase Storage，还是未来云上方案，属于部署实现层决策。

## 9. 核心业务流

### 9.1 录入流

1. 用户在 PWA 中上传截图 / 输入文字 / 拍名片
2. 系统保存原始内容
3. 系统提取结构化信息
4. 识别结果进入待确认队列
5. 用户确认 / 修改 / 跳过
6. 系统生成正式事件
7. 系统刷新客户最新跟进状态
8. 系统生成或更新任务

### 9.2 查看流

1. 用户打开客户列表
2. 搜索或进入某个客户
3. 查看基础信息、时间线、原始证据、任务和当前跟进状态

### 9.3 跟进流

1. 系统根据记录提取下一步
2. 若存在明确承诺，生成任务
3. 若没有明确时间，按分层提醒策略推导建议跟进时间
4. 用户可手动补任务或完成任务

## 10. 成功标准

第一版成功不看“功能数量”，而看下面这些是否成立：

- Chao 愿意持续在真实场景中使用
- 单条记录足够顺手
- 新记录能稳定归到正确客户名下
- 每次新增正式记录后，跟进状态会刷新
- 待跟进任务真的能减少遗漏
- 需要找历史沟通时，明显比手翻聊天记录快
