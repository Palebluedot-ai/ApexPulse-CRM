# 上线加速与需求澄清清单

## 1. 目的

这份文档定义当前从 M1.22 走向可真实使用上线前，需要澄清的问题、短中长期目标，以及在赶进度时如何保持质量。

这份文档是自包含文档。

阅读这份文档不需要依赖其他文档。

当前项目目标不是做一个完整通用 CRM。

当前项目目标是：

```text
让 Chao 能尽快在手机上真实记录微信截图、Review、入库、跟进，并持续使用。
```

## 2. 当前判断

当前已经不适合继续无限补需求。

当前应该进入上线倒计时模式：

```text
收窄范围 -> 固定金路径 -> 手机真实 dogfood -> 修关键问题 -> 小范围上线
```

当前最重要的不是再讨论很多未来功能。

当前最重要的是让第一条真实使用路径稳定：

```text
手机上传微信截图 -> AI 提取 -> 人工确认 -> 客户更新 -> 任务生成 -> 本周回顾
```

## 3. 需要 Chao 澄清的问题

以下问题会影响上线节奏或后续难改的架构。

### 3.1 阻塞上线的问题

这些问题建议尽快确认。

#### 3.1.1 第一版上线环境

需要确认：

```text
第一版真实使用先跑在 Mac 本机，还是直接上 Vercel / Supabase？
```

推荐默认：

```text
先 Mac 本机跑通真实使用，再上云。
```

原因：

- 当前最重要的是产品闭环，不是部署工程
- Mac 本机成本最低
- 数据和图片都在本地，调试最快
- 真实使用一周后再决定云部署更稳

如果 Chao 明确要外网随时访问，再切到云部署。

#### 3.1.2 手机访问方式

需要确认：

```text
第一轮手机 dogfood 能不能接受同 Wi-Fi 内网访问？
```

推荐默认：

```text
可以接受。
```

原因：

- 速度最快
- 不需要公网域名
- 不需要 HTTPS
- 不需要处理云存储

限制：

- 手机和 Mac 必须在同一个网络
- 离开本地网络后不能访问

#### 3.1.3 第一批真实数据规模

需要确认：

```text
第一轮真实 dogfood 先测几张微信截图？
```

推荐默认：

```text
第一轮 1 张，第二轮 3 到 5 张，第三轮 10 到 20 张。
```

原因：

- 当前还在校准 AI 提取质量
- 太多真实记录会增加清理成本
- 先验证路径，再验证批量体验

#### 3.1.4 上线前是否允许暂时保留开发默认密码

需要确认：

```text
第一版真实使用前，是否必须把 local-dev-password 换成 Chao 自己的密码？
```

推荐默认：

```text
必须换。
```

原因：

- 这是低成本安全动作
- 不影响开发速度
- 避免手机或局域网访问时使用公开默认密码

#### 3.1.5 真实数据删除策略

需要确认：

```text
如果 AI 识别错、创建了错误客户，第一版怎么处理？
```

推荐默认：

```text
第一版先不做删除，只做跳过、修正和后续合并/归档排期。
```

原因：

- 删除真实数据风险高
- 客户合并和删除都需要单独设计
- 短期可以通过人工修正避免继续污染

### 3.2 不阻塞上线的问题

这些问题可以后置。

- Leader / 下级关系怎么做
- 团队权限细节
- 云部署最终用 Vercel、Supabase、AWS 还是公司服务器
- 客户分层规则
- 客户合并规则
- 自动生成销售回复草稿
- 更复杂的 report
- PWA 离线能力

这些问题重要，但不应该阻塞第一版真实使用。

## 4. 短期目标

短期目标定义为接下来 1 到 3 天。

短期目标是：

```text
让 Chao 可以在手机上稳定跑 1 到 5 张真实微信截图。
```

短期开发目标：

1. 完成 M1.22A 手机端 dogfood runbook
2. 完成 M1.22B Capture 连续上传体验
3. 完成 M1.22C Review 连续处理安全带
4. 修正 README 当前状态
5. 配置本地真实密码
6. 跑 1 张真实截图手机端 E2E
7. 跑 3 到 5 张真实截图连续 Review

短期不做：

- 不做团队权限
- 不做云部署
- 不做自动入库
- 不做批量确认
- 不做客户合并
- 不做复杂数据清理

短期验收标准：

- 手机能打开 CRM
- 手机能登录
- 手机能上传截图
- Review 能 AI 提取字段
- 人工修正能确认入库
- 客户详情能看到最新沟通
- 任务页能看到 follow-up task
- `pnpm check` 通过
- `pnpm build` 通过

## 5. 中期目标

中期目标定义为接下来 1 到 2 周。

中期目标是：

```text
让 Chao 能每天真实使用，并且数据不会因为本地运行而容易丢。
```

中期开发目标：

1. 备份策略
2. 简单导出
3. 更清晰的错误处理
4. 真实截图连续处理体验优化
5. Weekly report 可读性优化
6. 客户详情第一屏继续打磨
7. 手机端 UI polish
8. 部署方案选择

中期需要决定：

- 继续 Mac mini 本地跑，还是上 Vercel / Supabase
- 图片存储继续本地，还是迁移到云存储
- 是否需要公网访问
- 是否需要每天自动备份

中期不建议做：

- 不建议马上做团队权限
- 不建议马上做复杂分层
- 不建议马上做自动化销售建议

原因：

真实个人使用一周后，这些需求会更清楚。

## 6. 长期目标

长期目标定义为 1 个月以后。

长期目标分三条线。

### 6.1 团队线

目标：

- Leader 可以添加下级
- Leader 可以看下级跟进情况
- 下级默认只看自己的客户和任务
- Report 支持个人和团队视角

风险：

- 权限设计错了后面很难改
- 管理视图可能影响团队沟通方式
- 需要明确哪些行为是监督，哪些是协作

### 6.2 AI 线

目标：

- 更稳定的截图识别
- 更稳定的客户名判断
- 更好的下一步提取
- AI 提供 follow-up 建议但不自动发送
- AI 生成 weekly summary 草稿

风险：

- AI 误判会影响销售动作
- 成本需要控制
- 必须保留人工确认

### 6.3 部署和产品化线

目标：

- 云部署
- 自动备份
- 稳定登录
- 文件存储迁移
- 错误监控
- PWA 安装体验
- 小团队长期可用

风险：

- 过早工程化会拖慢当前上线
- 上云后数据迁移和密钥管理要更谨慎

## 7. 赶上线时如何提速

当前提速原则：

```text
砍范围，不砍质量。
```

具体做法如下。

### 7.1 固定一条金路径

只优化这条路径：

```text
手机上传微信截图 -> Review -> AI 提取 -> 人工确认 -> 客户详情 -> 任务
```

其他路径先不追求完美。

好处：

- 测试范围变小
- 反馈更快
- 可以更快上线真实使用

### 7.2 使用上线闸门

每次进入下一小阶段前必须过闸门。

闸门包括：

- `pnpm check`
- `pnpm build`
- 手机端手工跑 1 条真实 case
- 不提交 `.env.local`
- 不提交 `data/`
- 不删除真实数据

这些不是拖慢速度。

这些是在防止返工。

### 7.3 并行推进文档和代码

开发时只写必要文档：

- 当前执行目标
- 系统日志
- 上线 checklist

不再为每个小 UI 改动写长篇 PRD。

原因：

- 当前已经过了大方向澄清阶段
- 现在重点是落地和验收

### 7.4 用默认决策减少等待

以下事项默认由 Codex 决策，不再每次问 Chao：

- 按钮文案
- 页面小布局
- 错误提示文案
- 测试文件组织
- 非破坏性重构
- 不改变产品边界的 UI polish

以下事项必须问 Chao：

- 删除真实数据
- 重做 migration
- drop table
- 改部署方向
- 改成自动入库
- 改成团队优先
- 改权限模型
- 真实 API key 或真实密钥变更

### 7.5 小步提交

每个可验证小闭环都可以单独 commit。

推荐节奏：

1. M1.22A 文档和 runbook
2. M1.22B Capture 上传体验
3. M1.22C Review 连续处理
4. M1.22D README 和上线 checklist
5. 手机真实 dogfood 修正

这样如果某一步出问题，可以回滚单点，不影响全部进度。

## 8. 当前推荐下一步

下一步不再继续大范围需求讨论。

下一步直接执行：

```text
M1.22A + M1.22B
```

也就是：

- 补手机端 dogfood runbook
- 优化 Capture 上传后的连续录入体验
- 保持一张图一个 pending review event
- 不做复杂批量上传

完成后立刻做：

```text
手机上传 1 张真实微信截图的 E2E 测试
```

如果这条跑通，再做：

```text
连续 3 到 5 张真实微信截图测试
```

这就是最快、也最不容易失控的上线路径。
