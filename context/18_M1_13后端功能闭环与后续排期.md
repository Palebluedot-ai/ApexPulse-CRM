# M1.13 后端功能闭环与后续排期

## 1. 文档目的

这份文档记录 2026-05-29 晚上 Chao 对后续功能闭环的最新要求。

这份文档是自包含的。读这份文档不需要先读其他文档。

## 2. 最新方向

当前优先级从“图片预览体验 polish”调整为“先完成所有功能闭环，特别是后端”。

这意味着：

- 不先追求界面细节
- 不先做 OCR / Vision API
- 不先做团队权限
- 不先做复杂 report
- 先把 Review 确认后的客户、事件、任务闭环补完整

## 3. 当前已完成基础能力

当前系统已经具备：

- 文字录入
- 图片真实本地上传
- 图片附件本地保存
- 待确认队列
- Review 自然字段确认
- 绑定已有客户
- 确认后刷新已有客户的最新沟通状态
- 手动任务创建
- 任务完成 / 恢复

## 4. 当前缺口

当前最大缺口有两个。

### 4.1 新客户闭环缺口

当 Review 页面选择“新客户 / 暂不匹配现有客户”时：

- 如果用户填写了客户名，系统还不会自动创建客户
- 事件不会自动绑定到新客户
- 公司、来源、需求摘要不会沉淀到客户档案

这会导致新 lead 无法真正进入 CRM。

### 4.2 下一步任务闭环缺口

当 Review 页面填写“下一步”和“下次跟进时间”时：

- 系统还不会自动生成任务
- 下次跟进时间还没有变成 task dueAt
- 用户仍然要手动再建一次任务

这会导致 follow-up 动作没有自动落地。

## 5. M1.13 推荐目标

M1.13 建议定义为：

`Review 确认后的后端功能闭环`

M1.13 目标：

- Review 确认时可以自动创建新客户
- Review 确认时可以自动生成跟进任务
- 事件正确绑定客户
- 客户最新沟通状态继续刷新
- 任务可在任务页看到、完成和恢复

## 6. M1.13 建议实现范围

### 6.1 新客户创建

当用户确认 Review 记录时：

- 如果选择了已有客户，就绑定已有客户
- 如果没有选择已有客户，但填写了客户名，就创建新客户
- 如果没有选择已有客户，也没有填写客户名，就只确认事件，不创建客户

新客户字段建议：

- `displayName` 来自客户名
- `companyName` 来自公司
- `referralSourceTag` 来自来源
- `profileSummary` 可先来自需求摘要
- `followupStatus` 由现有跟进状态逻辑决定

第一版不做复杂去重。

如果同名客户重复，先允许重复。

原因：

去重、合并、模糊匹配会引入额外产品规则，不应该混进第一版后端闭环。

### 6.2 自动任务生成

当用户确认 Review 记录时：

- 如果填写了“下一步”，系统自动创建一条 followup 任务
- 如果填写了“下次跟进时间”，任务 dueAt 使用该时间
- 如果没有填写“下一步”，不自动生成任务

任务字段建议：

- `partyId` 绑定当前确认后的客户
- `sourceEventId` 绑定当前事件
- `taskType` 为 `followup`
- `description` 来自“下一步”
- `dueAt` 来自“下次跟进时间”
- `status` 为 `open`

### 6.3 不做的事情

M1.13 不做：

- 客户去重
- 客户合并
- 复杂分层规则
- 团队权限
- 周报
- OCR / Vision API
- 登录系统

## 7. 后续功能排期

### 7.1 M1.14 登录验证

Chao 新增要求：

每个人都需要登录。

当前建议：

- 本地 dogfood 阶段可以先做最小登录保护
- 首版不做复杂权限
- 先保证每个用户有身份
- 后续团队能力再使用这个身份体系

M1.14 可能范围：

- 登录页
- session
- 当前用户识别
- createdBy / reviewedBy / owner 字段开始接真实用户

### 7.2 M1.15 Report / Weekly To-do

Chao 新增需求：

- 本周 report
- 本周 to-do list
- 记录这一周跟进了谁
- 记录这一周做了什么

建议先做只读报告。

第一版 report 可以包括：

- 本周新增客户数
- 本周确认沟通数
- 本周完成任务数
- 本周未完成任务
- 本周跟进过的客户
- 每个客户最近一次沟通摘要

第一版不需要 AI 生成长文。

先让后端聚合数据准确。

### 7.3 M2 Leader / 下级关系

Chao 新增方向：

- Leader 可以添加自己的下级
- Leader 可以查看下级所有跟进情况

当前建议放入 M2。

原因：

- 会影响权限模型
- 会影响数据可见性
- 会影响用户管理
- 会显著增加验证复杂度

M2 前建议单独做一次 Office Hours 澄清。

需要确认：

- 下级能不能看到 Leader 的客户
- 平级之间是否完全隔离
- Leader 是否能编辑下级客户
- 离职或转组时客户归属如何处理
- Report 是按个人、团队还是客户维度汇总

## 8. 当前推荐开发顺序

当前建议顺序：

1. M1.13 Review 确认后自动创建新客户
2. M1.13 Review 确认后自动生成任务
3. M1.14 登录验证
4. M1.15 Weekly report / to-do list
5. M1.16 图片预览体验 polish
6. M2 Leader / 下级关系
7. M2 OCR / Vision API

## 9. 当前结论

下一步不应该先做界面美化。

下一步应该先补 Review 确认后的后端闭环。

只要新客户和下一步任务能自动落地，这个 CRM 才真正从“记录系统”进入“跟进系统”。
