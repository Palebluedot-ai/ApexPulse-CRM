### 2.1 已创建内容

### 2.2 技术选择

- 前端框架：Next.js 16
- UI 方向：PWA-first
- 样式：Tailwind CSS 4
- 语言：TypeScript
- 测试：Vitest
- 包管理：pnpm

### 2.3 为什么这样做

Next.js App Router 适合先做 PWA，再逐步增加服务端能力。

Tailwind CSS 适合快速搭建第一版界面，不需要先投入复杂设计系统。

Vitest 适合测试后面的业务规则，例如跟进状态刷新、任务生成、review 状态流转。

pnpm 安装速度快，也能让依赖版本更稳定。

这代表：

- typecheck 通过
- test 通过

`pnpm build` 已通过。

这代表：

- PWA manifest 可以静态生成

本地 dev server 已验证：

### 2.5 发现并处理的问题

第一次安装依赖时，`eslint@10` 与 Next 生态依赖链存在 peer dependency 风险。

处理方式：

- 固定 Next / React / Tailwind / Vitest 等版本
- 把 ESLint 固定在 Next 生态明确支持的 9.x

第一次生产构建在普通沙盒里失败。

原因：

- Turbopack 构建 CSS 时需要创建子进程并绑定本地端口
- 普通沙盒不允许这个动作

处理方式：

- 使用受控提权重新构建
- 构建通过，说明不是代码问题

Next.js 第一次构建时提示匿名 telemetry。

处理方式：

- 在 `dev`、`build`、`start` 脚本里默认设置 `NEXT_TELEMETRY_DISABLED=1`

### 8.2 已落地规则

待确认队列只读取 `review_status = pending_review` 的事件。

编辑待确认事件会：

- 更新 `events.ai_summary`
- 更新 `events.extracted_fields_json`
- 保持 `review_status = pending_review`
- 不删除原始事件
- 不删除附件

跳过待确认事件会：

- 设置 `review_status = skipped`
- 不删除原始事件
- 不删除附件

确认待确认事件会：

- 设置 `review_status = confirmed`
- 更新 `events.ai_summary`
- 更新 `events.extracted_fields_json`
- 可选更新 `events.occurred_at`
- 可选绑定 `events.party_id`

在 Step 3 这个小闭环中，确认事件只负责事件自身的 review 状态变化。

这个边界是刻意保留的，因为客户字段刷新属于独立小闭环：

`Step 4：确认后刷新客户状态`
